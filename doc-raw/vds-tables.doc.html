Layer stack

The tables subsystem is token-driven and depends on primitives, base, layout, and one active theme. Load CSS in this exact order before using any .table* classes.

  • primitives.css — all design tokens (spacing, radii, colour primitives).
  • base.css — global reset, typography baseline, element defaults.
  • layout.css — containers, grid system, section spacing.
  • content-blocks.css — optional structural blocks used in documentation.
  • doc-block.css — documentation formatting (preview / code blocks).
  • tables.css — the tables subsystem (this file).
  • icons.css — optional icon set (trend arrows do not require it).
  • One active theme from /themes/*.css — colours, surfaces, semantic hues.

Accessibility & usage rules

Tables require strict structural semantics, predictable reading order, numeric clarity, responsive fallbacks and assistive-friendly labels. VDS tables are semantic-first, token-driven and support complex data (finance, metrics, trends) without breaking screen-reader parsing or mobile usability.

Do

  • Use semantic table structure: <table>, <caption>, <thead>, <tbody>, <th scope="col">, <th scope="row">.
  • Provide a caption for every data table — essential for screen-reader context.
  • Keep headers sticky (.table--sticky-header) only when the table is long enough; sticky elements remain semantic headers.
  • Use the VDS numeric helpers correctly: .t-num, .t-cur, .t-pct, .t-trend, .t-up, .t-down, .t-flat.
  • For currencies (.t-cur), rely on the theme-driven symbol via ::before (€, $, £, RON, CHF, ¥).
  • For percentage and numeric values, ensure tabular numerals are preserved (font-variant-numeric: tabular-nums).
  • In responsive collapse mode (.table--collapse), ensure each td includes a data-label matching the column header.
  • Use density controls (.table--a/--b/--c or .table--compact) to improve readability based on content complexity.
  • Use .table-panel wrappers for grouped contexts with proper header labeling (.table-panel__title).
  • Use .table-panel__actions to group filters, exports, or bulk actions without stretching the header.
  • Apply row-state classes semantically: .table-row--success, --danger, --warning, --info, --muted.
  • Use .table--fixed with .table-cell--truncate when you need ellipsis for long text, and expose full values in title attributes.
  • Add a totals row in <tfoot> (or .table-row--summary) for financial summaries.
  • Use expand/collapse rows (.table-row--expandable → .table-row--details) with clear JS-controlled semantics and visible labels.
  • Make sure all action icons in .table-actions have tooltips or sr-labels when icon-only.
  • Use finance states (namespaced .table-fin--good/.table-fin--bad/.table-fin--warning/.table-fin--info/.table-fin--muted/.table-fin--flagged/.table-fin--curious) for meaning — good / bad / flagged, etc. — not decoration.
  • Loading rows (.table-loading) must remain semantic rows — the shimmer overlay affects only visual rendering.

Don’t

  • Don’t fake tables with <div> grids — assistive tech relies heavily on table semantics.
  • Don’t reorder columns visually using CSS; numeric meaning depends on consistent left→right reading order.
  • Don’t use ellipsis truncation (.table--truncate-cells) for critical fields like names, IDs, or categories.
  • Don’t combine contradictory variants (.table--bordered + .table--clear, etc.).
  • Don’t use row states as decorative colour stripes — they imply semantic meaning.
  • Don’t remove caption in complex or financial tables.
  • Don’t rely solely on colour to convey trend direction — pair trend helpers with explicit text (“+4.3%”, “–2.1%”).
  • Don’t override header font-weight, border logic or cell padding via inline CSS — these are token-driven and system consistent.
  • Don’t use sticky columns (.table--sticky-col) unless the table horizontally scrolls; otherwise it creates unnecessary overlap.
  • Don’t expect .table-cell--truncate to work without a fixed layout (.table--fixed) or explicit column widths.
  • Don’t forget data-label for collapsed mobile mode — missing labels break screen-reader comprehension.
  • Don’t hide entire rows to simulate a collapse — use the official expand/collapse pattern instead.
  • Don’t place interactive elements inside <th> unless explicitly labelled (sort buttons, filters, etc.). 
1. Semantic structure mandatory (table/thead/tbody/th scope).
2. Every table needs a caption for assistive context.
3. Numeric cells require tabular-nums and correct alignment.
4. Currency uses span.t-cur with theme-driven symbol.
5. Trends use t-trend + arrow states; pair colour with text.
6. Collapse mode requires data-label on every td.
7. Sticky headers/cols only for scrollable scenarios.
8. Expandable rows must remain semantic rows.
9. Row states and finance states carry meaning; not decoration.
10. Don’t override cell padding, font-size, or borders inline.
11. Table actions must provide text or sr-only labels.
12. Loading states stay semantic; shimmer is visual only.
13. Mobile collapse = structural rewrite; never simulate via visibility.
