Base Layer

base.css establishes the universal foundation for the Vlah Design System. It applies a modern hybrid reset, defines safe HTML defaults, installs the neutral token layer, and prepares the browser environment for consistent typography, layout, accessibility, and theme behaviour.

The base layer has no colours, no layout rules, and no component styling. It is an intentionally minimal, predictable surface that ensures every other CSS layer behaves consistently across browsers and devices.

Required Load Order

  1. primitives.css (neutral tokens)
  2. theme.css (slate / navy / carbon)
  3. base.css (consumes tokens)

This ensures that all colour and typography tokens exist before the base layer references them.

Base Philosophy — Hybrid Reset

VDS does not use a full “reset.css” or a full “normalize.css”. Instead, it applies a hybrid reset: removing unsafe defaults while preserving useful, expected browser behaviour.

Note: base.css does not define colours but does consume theme-level colour tokens (e.g., background, text, border). Themes must load before base.css so these tokens exist when used.

Note: The base layer applies minimal structural styling to <pre> elements — specifically background and border surfaces — using theme semantic tokens (--code-bg, --code-border). These are not aesthetic styles but foundational surfaces expected by documentation, monospace, and developer-focused components.

Note: Link colours are not defined by base.css; they are provided by the active theme. Base.css consumes these tokens to unify behaviour but does not impose visual design.

What Hybrid Reset means

  • Preserves natural HTML structure (headings, lists, form controls)
  • Normalizes cross-browser inconsistencies (line-height, box-model, focus)
  • Removes only unsafe defaults (margins, tap delays, inconsistent font inheritance)
  • Ensures predictable typography baseline
  • Installs system tokens before any theme or component layer loads
  • Avoids opinionated visual styling — colours and layout belong elsewhere

Rationale

An over-aggressive reset forces every design system to rebuild native behaviours from scratch. A full normalize is too passive. VDS occupies the middle ground: a surgical, modernized foundation designed specifically for token-driven UI systems.

HTML Defaults

The base layer defines predictable defaults for html, body, typography inheritance, scroll behaviour, and interaction surfaces. These defaults ensure all components render consistently.

html

The html element sets the global line-height, system font-smoothing, and the base font-family (inherited by everything unless overridden in typography). 

html {
    -webkit-font-smoothing: antialiased;
    -moz-osx-font-smoothing: grayscale;
}

body

The body’s responsibility is environmental: overflow, default background, and removing built-in spacing.

HTML 
body {
    margin: 0;
    min-height: 100vh;
    display: flex;
    flex-direction: column;
    text-rendering: optimizeLegibility;
}

VDS uses a body-as-flex-root model to guarantee a stable page shell, sticky footer behaviour, and consistent app-like layouts. This is a deliberate structural choice at the base layer.

Lists

Lists remain semantically intact. No bullets are removed; no padding is stripped. Components or utilities control appearance when needed.

HTML 
ul, ol {
    padding-inline-start: var(--space-6);
}

The native list indentation is preserved conceptually, but the value is normalized through the spacing token system rather than a browser default or static pixel/rem value.

Images & media

Images become responsive by default, preventing layout overflow without forcing aspect ratios.

HTML 
img,
picture,
video,
canvas,
svg {
    display: block;
    max-width: 100%;
}

Forms

Form controls inherit typography but avoid opinionated styling. Interaction defaults are normalized across browsers.

HTML 
button,
input,
textarea,
select {
    font: inherit;
    color: inherit;
    margin: 0;
}

Global Structure & SPA Roots

Base installs a predictable page flex root for both traditional documents and single-page applications. Layout layers build on this; they do not redefine it.

Document flow

HTML 
html,
body {
    width: 100%;
}

body {
    display: flex;
    flex-direction: column;
}

main {
    flex: 1 0 auto;
}

footer {
    flex-shrink: 0;
}

The document body is a vertical flex container. main grows to fill the viewport, while footer remains pinned to the bottom without absolute positioning. This ensures predictable full-height layouts on every page.

SPA root containers

HTML 
#root,          /* React, Vite */
#app,           /* Vue, Svelte, Solid */
#__next,        /* Next.js */
#__nuxt,        /* Nuxt */
app-root,       /* Angular */
svelte-root,    /* Some Svelte starter kits */
astro-root,     /* Astro hybrid apps */
[qwik-root],    /* Qwik */
.app-root,      /* Generic container */
[data-app-root],/* Custom micro-frontend root */
[data-spa-root] /* Universal custom attribute */
{
    display: flex;
    flex-direction: column;
    min-height: 100vh;
    width: 100%;
}

All known SPA roots are normalized as flex column containers with min-height: 100vh. Framework shells can rely on this behaviour and avoid redefining “page-height” boilerplate in every project.

Layout belongs to layout.css. Base only guarantees that the root containers behave like full-height flex pages. Sections, content max-widths and responsive grids live in the layout layer.

Interaction & Input Normalization

VDS normalizes interactive elements to avoid browser inconsistencies in focus rings, button appearance, and text rendering. Component layers provide the visual styling.

Buttons

HTML buttons often have UA-specific padding and borders. The base layer removes these without touching the visual style.

HTML 
button {
    border: none;
    background: none;
    padding: 0;
}

Focus-visible

Focus logic is unified across browsers. Visual styling is defined in utilities.

HTML 
:focus {
    outline: var(--focus-ring-width) solid var(--focus-ring-color);
    outline-offset: var(--focus-ring-offset);
}
:focus:not(:focus-visible) {
    outline: none;
}

:focus-visible {
    outline: var(--focus-ring-width) solid var(--focus-ring-color);
    outline-offset: var(--focus-ring-offset);
}

Selection

Selection colour is not set by base. Themes may override if desired.

HTML 
::selection {
    background: var(--color-selection-bg);
    color: var(--color-selection-text);
}

Root sizing model

VDS uses a root-based rem model. All spacing, typography and layout primitives are defined in rem, driven by the browser’s default font size. The base layer does not override the user’s root font-size preference.

Rem philosophy

  • 1rem = browser default (usually 16px)
  • No hard-coded pixel sizes in base layer
  • Tokens express rem values; components consume tokens
  • User zoom & accessibility settings remain respected

This means that any user who adjusts their browser default font-size automatically scales the entire system. No additional logic is needed in layout or typography.

HTML 
/* Base does NOT fix html font-size to 62.5% or similar.
   It keeps the user’s default and lets tokens operate in rem. */

html {
    /* no explicit font-size here */
}

Any explicit rem mapping (e.g. 0.25rem → 4px) is documented in the token layer, not enforced at the base level.

Typography baseline & inheritance

The typography system lives in its own file, but the base layer ensures that all elements inherit fonts and sizing predictably, without UA-specific deviations.

Body as typographic root

body acts as the default typographic root for the document. Component layers and typography utilities build on this, never bypass it.

HTML 

    font-family: var(--font-family-sans), sans-serif;
    font-size: var(--text-base);
}

All headings, paragraphs, form controls and basic text elements inherit from body unless explicitly overridden by typography classes.

Monospace & code

Monospace contexts are normalized to prevent user agent variations. The base layer sets a token-driven background, border and text colour for block code via pre, but leaves inline code visually neutral.

HTML 
code {
    font-family: var(--font-family-mono), monospace;
    font-size: 0.875em;
}

pre {
    font-family: var(--font-family-mono), monospace;
    font-size: 0.875em;
    background: var(--code-bg);
    color: var(--color-text);
    border: 1px solid var(--code-border);
    overflow: auto;
    white-space: pre-wrap;
    word-break: break-word;
}

pre, code {
    tab-size: 4;
}

kbd,
samp {
    font-family: var(--font-family-mono), monospace;
}

Component and typography layers add any additional code styling (inline code backgrounds, padding, syntax themes). Base only establishes a predictable monospace baseline.

Box-model & inline normalization

Base ensures that both block and inline elements participate in layout in a predictable way: margins are neutralized, and the box-model is consistent across the tree.

Neutral Token Layer

Neutral tokens are foundational geometry and motion primitives defined in primitives.css and consumed by base.css; the base layer requires these tokens but does not declare them. They contain no colours and no component behaviour. These tokens are inherited by typography, layout, utilities, and components without exception.

The base token layer ensures that spacing, radius, borders, and motion remain consistent across all components before themes or component styling are applied. These tokens represent the non-negotiable core API of the Vlah Design System.

Links & Interaction Defaults

Links maintain semantic behaviour and accessibility, while their decoration rules are unified. Base does not assign colours — themes do.

Link behaviour

By default, links preserve their underline and inherit text colour, ensuring consistency across themes. This is appearance-neutral.

    color: var(--link-text, inherit);
    text-decoration: none;

    color: var(--link-text-hover, inherit);

    color: var(--link-text-active, inherit);

Hover only adjusts underline thickness; no colour changes occur in base.

Active / visited states

Base does not define explicit styles for :visited or :active. These states are theme-level responsibilities.

Accessibility Primitives

The base layer installs predictable accessibility behaviour: focus-visible, reduced-motion, system-specific improvements, and scroll-behaviour logic.

Focus-visible

HTML 
:focus {
    outline: var(--focus-ring-width) solid var(--focus-ring-color);
    outline-offset: var(--focus-ring-offset);
}
:focus:not(:focus-visible) {
    outline: none;
}

Removes “always-on” outlines without removing keyboard focus indicator.

Reduced motion

HTML 
@media (prefers-reduced-motion: reduce) {
    :root {
        --transition-fast:   var(--motion-none);
        --transition-normal: var(--motion-none);
        --transition-slow:   var(--motion-none);

        /* Optional: linear easing under reduced motion */
        --ease-out:    linear;
        --ease-in:     linear;
        --ease-in-out: linear;
    }

    /* Safety-net for non-token transitions */
    * {
        animation-duration: 0ms !important;
        animation-iteration-count: 1 !important;
        transition-duration: 0ms !important;
        scroll-behavior: auto !important;
    }

    html {
        scroll-behavior: auto;
    }
}

This is a critical part of VDS accessibility: animation, transitions, and smooth scrolling are effectively disabled.

Base-Layer Invariants

Base-layer invariants are rules that MUST NOT change. They ensure that typography, layout, utilities, and all components behave predictably across devices, browsers, and themes.

1. The browser's root font-size is never overridden.
VDS respects user accessibipty settings. All scapng uses rem.

2. box-sizing is border-box for every element.
All layout, utipties, and components depend on this invariant.

3. The base layer defines NO colours.
All colours come from themes. Base never includes background, text colours, borders, shadows.

4. No component stypng is allowed in base.
Buttons, inputs, cards, navigation, overlays — all belong to their subsystems.

5. Headings inherit typography.
No heading sizes are defined in base. Typography utipties control scale.

6. Form controls inherit font and colour.
Base strips UA chrome but does not style inputs.

7. Margin reset is global and permanent.
All vertical rhythm is driven by section/stack patterns.

8. Scroll behaviour is neutral by default.
Base does not enforce smooth scrolling. It only forces scroll-behavior: auto under prefers-reduced-motion: reduce as an accessibility guarantee.

9. Reduced-motion overrides must remain strict.
Accessibipty comes before animation.

10. No full reset, no normapze — hybrid reset only.
VDS preserves semantics, corrects inconsistencies, and avoids destructive resets.

Anti-Patterns

Incorrect uses of the base layer that destabilize the design system.

Anti-pattern 1 — Adding colour tokens to base

Base must stay neutral. Adding colour variables here makes themes impossible to swap or extend.

HTML 
/* WRONG — DO NOT DO THIS */
:root {
    --color-brand: #00aaff;
}

Anti-pattern 2 — Styling buttons or inputs in base

Button/input visuals belong to Forms and Buttons. Base only removes UA chrome.

HTML 
/* WRONG — DO NOT DO THIS */
button {
    padding: 0.5rem 1rem;
    background: #eee;
}

Anti-pattern 3 — Heading size overrides

Size and line-height belong to typography scale, not base.

HTML 
/* WRONG — DO NOT DO */
h1 {
    font-size: 3rem;
}

Anti-pattern 4 — Reintroducing default margins

Reintroducing `margin-bottom` on p or h1 breaks the entire vertical-rhythm engine.

HTML 
/* WRONG — DO NOT DO */
p {
    margin-bottom: 1rem;
}

Anti-pattern 5 — Overriding box-sizing

The entire spacing and layout system depends on border-box.

HTML 
/* WRONG */
* {
    box-sizing: content-box;
}

Anti-pattern 6 — Adding layout rules to base

Layout belongs strictly to layout.css and utilities.

HTML 
/* WRONG */
main {
    max-width: 80rem;
    margin: 0 auto;
}

Accessibility & usage rules

The base layer establishes the non-negotiable accessibility contract of the VDS: predictable semantics, preserved user settings, reduced-motion enforcement, and safe HTML defaults. It must remain colour-neutral, layout-neutral, and free of component logic.

Do

  • Preserve user font-size and zoom preferences: html must not set an explicit font-size.
  • Respect prefers-reduced-motion — base.css already zeroes animation, transitions and smooth scrolling, and this behaviour must never be weakened.
  • Keep box-sizing: border-box universal — the spacing and layout engines rely fully on this invariant.
  • Maintain inherited typography for all form controls: buttons, inputs, selects, and textareas must use font: inherit.
  • Treat block code (<pre>) as a foundational readability surface: its background and border are semantic surfaces, not design.
  • Ensure link colour tokens exist in themes; base consumes them but does not define them.
  • Preserve semantic HTML: native headings, lists, paragraphs and figures should not be visually styled at the base layer beyond reset/normalization.
  • Rely on sections/stacks in layout.css for vertical rhythm — margins remain neutral in base.
  • Ensure SPA roots (#root, #app, #__next, …) remain full-height flex containers capable of driving consistent shells.

Don’t

  • Don’t add colour variables to base.css — all colours belong strictly to theme layers.
  • Don’t introduce padding, border, shadows, or component visuals — base must remain appearance-neutral.
  • Don’t override headings, font-weight, line-height, or text sizes — typography.css controls scale.
  • Don’t reintroduce margins on h1, p, ul, ol or other block elements. Vertical rhythm is not handled here.
  • Don’t override UA behaviour that base normalizes — especially scrolling, interaction defaults, or focus-visible.
  • Don’t hide number-input spinners or style search inputs — such decisions belong to the Forms/Buttons subsystems.
  • Don’t attach layout rules to main, footer, or SPA roots — layout.css handles spacing and max-widths.
  • Don’t weaken or remove reduced-motion enforcement — accessibility takes priority over animation fidelity. 
1. Never set html font-size (respect user scaling).
2. Universal border-box is mandatory.
3. Reset margins globally; layout primitives define rhythm.
4. Form controls inherit typography and colour.
5. Link colours come from themes; base is appearance-neutral.
6. Code blocks (<pre>) get token-based surfaces only.
7. Reduced-motion disables transitions/animations system-wide.
8. No component visuals allowed in base.
9. SPA roots must remain full-height flex containers.
10. No colour tokens, no layout rules, no component logic in base.
