VLAH DESIGN SYSTEM (VDS)

A modular, token-driven design system built for clarity, durability, and production-grade engineering across all VLAH Group digital properties. VDS provides identity, layout, components, themes, utilities, and documentation tooling — all structured as independent layers with strict contracts.

This landing page introduces the full architecture of VDS, explains how to navigate this documentation, and outlines the expectations, guarantees, and structure of each subsystem.

Everything here is built from first principles: spacing tokens, color primitives, theme layers, structural layout rules, semantic density, components, and documentation tooling. VDS is deterministic: every file binds to a purpose, exposes a clear contract, and extends the system without mutation.

System Architecture

VDS is organised in layers. Each layer has a precise responsibility, guaranteed stability rules, and zero side-effects on adjacent layers.

Core Layers

  • Primitives: spacing, typography scales, radii, borders, shadows, transitions.
  • Themes: slate, navy, carbon; semantic tokens; dark/light pairs.
  • Base Layer: canonical resets, base elements, accessibility defaults.
  • Identity Layer: VLAH brand, logo marks, color locks, symbolic rules.
  • Layout Layer: containers, grids, sections, responsive structure.

Component Layers

  • Accordion, Buttons, Tables, Tabs
  • Badges & Tags, Avatars, Description Lists
  • Action Bar, Progress, Empty States
  • Forms & Advanced Forms
  • Navigation, Header & Footer
  • Content Blocks, Sections, Skeleton, Hero
  • Toasts, Tooltips, Popovers, Overlays
  • Icons, Charts, Command Palette
  • Feedback (alerts, messages, semantic feedback)
  • Guidance (education blocks and instructional panels)

Utility Layer

  • Spacing utilities
  • Flex utilities
  • Typography utilities
  • Color & surface utilities
  • Display, interactivity, density overrides

Documentation Layer

  • doc-block.css: documentation UI
  • authoring.css: prose-enhancement, footnotes, code blocks, callouts
  • examples + specimen: consistent block structure for demos

Documentation Navigation

Documentation pages are loaded dynamically into #doc-content. Click any item in the left/right footer navigation to load the corresponding vds-*.doc.html specification.

How to use this documentation

  • Click a topic in the footer navigation.
  • doc-loader.js injects the requested file into #doc-content.
  • Every topic is a self-contained file: examples + preview + code + rules.
  • All examples follow the doc-block structure for consistency.
  • Components never depend on documentation styles — production code stays clean.

Structure of each doc page

  • Introduction: what the component does.
  • Specimen: live UI examples.
  • Installation & dependencies: required CSS files.
  • Anatomy: exact class contract.
  • States & variants: semantic/density layers.
  • Usage rules: when to use, when not to use.
  • Accessibility: semantic behaviour & keyboard rules.

Using VDS in Production

This section describes how to include VDS in real products: minimal bundles, theme selection, component imports, and semantic guarantees.

Production Principles

  • Load only the CSS files required for your features.
  • Themes are interchangeable: slate, navy, carbon.
  • Components never modify base elements.
  • Utilities override locally — no global mutations.
  • Density classes (.density-a, .density-c) are opt-in.

Minimal viable bundle

<link rel="stylesheet" href="primitives.css">
<link rel="stylesheet" href="themes/slate.css">
<link rel="stylesheet" href="base.css">
<link rel="stylesheet" href="layout.css">
<link rel="stylesheet" href="utilities.css">
<link rel="stylesheet" href="components/buttons.css">
<link rel="stylesheet" href="components/forms.css">

Theme switching

You can toggle themes at runtime using the data-theme attribute on <html>:

<html data-theme="navy-dark">

The theme-switcher UI included in this documentation is optional and not required in production.
