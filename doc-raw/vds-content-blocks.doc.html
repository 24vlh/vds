Content Blocks

Structural UI primitives: cards, media blocks, lists, callouts, stats, tags, avatars, features, processes, highlights and fact blocks — all token-driven and theme-bound.

Source: content-blocks.css · Sits above primitives.css, themes/*.css, base.css and typography.css.

Installation & dependencies

Content blocks sit above the base and typography layers. They inherit spacing, radius, surface and motion tokens from the active theme and neutral primitives.

  • Load neutral tokens: primitives.css.
  • Load one active theme: themes/slate.css, themes/navy.css, or themes/carbon.css.
  • Load core layers: base.css, layout.css, typography.css.
  • Optionally load utilities, icons and buttons: utilities.css, icons.css, buttons.css.
  • Load content-blocks.css as the content shell layer.

Content-block anatomy

The content-block component is a neutral structural shell composed of up to four optional regions: header, media, body and footer. Each region is independent, token-aligned, and safe to mix or omit depending on layout needs.

content-block__header
Context label
Header region
Optional. Use for block titles, microcopy or short intro text.
content-block__media
content-block__body

The body is the primary content container. It defines vertical rhythm, respects typography scale, and adapts to all density modifiers.

content-block__footer

Region definitions

  • .content-block — root wrapper; establishes stack, spacing and vertical rhythm.
  • .content-block__header — optional top region for titles, meta, labels or short prose.
  • .content-block__eyebrow, .content-block__title, .content-block__subtitle — optional typographic slots for compact headings.
  • .content-block__meta, .content-block__actions — metadata and action groupings inside header, body or footer regions.
  • .content-block__media — optional media container for images, charts or icons (auto-centred).
  • .content-block__body — main content region; intended for prose, patterns, repeated blocks.
  • .content-block__footer — optional region for actions, secondary controls or meta links.
  • .content-block__main, .content-block__aside — split layout columns for primary and secondary content.
  • .content-block__grid, .content-block__list, .content-block__rows — structured layout helpers for repeated data.
  • .content-block__kv, .content-block__kpis — key-value and metric groups.

Variant behaviour

Content-block variants apply to the entire wrapper and affect padding, borders, radii and surface tone:

  • .content-block--surface — elevated card-like surface with border and padding tokens.
  • .content-block--inset — subtle surface with lower contrast and lighter padding.
  • .content-block--compact — reduced spacing and tighter layout rhythm.
  • .content-block--bordered — strong-outline variant for emphasis or separation.
  • .content-block--tint, .content-block--outline, .content-block--rail — semantic surface, outline and rail accents.
  • .content-block--hero, .content-block--center — larger padding and centered layouts for CTA blocks.
  • .content-block--split, .content-block--split-reverse — two-column layouts with main/aside regions.
  • .content-block--elevated, .content-block--flat — control shadow and elevation.
  • data-variant="info|success|warning|danger|accent|neutral" — injects semantic tokens for rails, tint, text and borders.
  • .content-block--interactive — hoverable/pressable shell with token-driven transitions.
  • .content-block--disabled — non-interactive, lower-opacity state with pointer suppression.

Patterns: card vs content-block vs callout

Cards are opinionated UI surfaces, content-blocks are neutral structural shells, and callouts are communication primitives. Use each pattern deliberately depending on emphasis, purpose and visual semantics.

card

Card

Cards are opinionated components: fixed padding, rounded surface, elevation and consistent content rhythm.

content-block

A content-block is a neutral structural wrapper. It defines regions but no visual opinion. Use for documentation bodies and tool surfaces.

Flexible footer
callout (highlight)
Callouts communicate.

Use callouts to draw attention to key information or warnings. They are stylistically strong and semantically expressive.

Role & usage

  • Card — UI component for product surfaces, dashboards, feeds, profile snippets, lists. Strong surface, fixed padding, structured regions.
  • Content-block — unopinionated container used in docs, marketing content, or to group related information without forcing styling.
  • Callout — communication element used to highlight, warn, or inform with semantic tone (info, success, warning, danger).

Comparison

Pattern Visual opinion Padding Structure Typical usage
card Strong Fixed, dense tokens Header, body, footer UI surfaces, dashboards, product blocks
content-block Neutral Variable (surface/inset/compact) Header, media, body, footer Docs, guides, marketing content, flexible compositions
callout / highlight High (semantic) Fixed Icon + body Communicating status or key information

Comparison: card vs content-block vs callout

Cards are opinionated UI surfaces. Content-blocks are neutral structural wrappers. Callouts highlight or communicate status. Choose based on purpose, emphasis and visual semantics.

Card

Opinionated. Fixed padding, surface, and radii. Ideal for UI patterns, dashboard elements, snippet blocks.
Neutral structural container. Define layout and rhythm without forcing a visual style. Suitable for docs and marketing.
Callout. Semantic surface for emphasis. Use for warnings, notes, tips or important messages.
Pattern Visual opinion Padding Structure Usage
card Strong Fixed Header, body, footer UI surfaces, dashboards, product blocks
content-block Neutral Variable Header, media, body, footer Docs, marketing, knowledge surfaces
callout Semantic Fixed Icon + body Highlighting information or alerts

Accessibility & usage rules

Contract for safe, semantic composition of cards, media blocks, list groups, callouts, stats, tags, avatars, feature/process/highlight/fact blocks. Maintain native HTML semantics, rely on token-driven interaction states, and avoid overloading these blocks with layout responsibility.

Do

  • Use native interactive elements (<button>, <a>) for interactive cards, list rows, chips and tags.
  • Provide accessible names for interactive blocks: use inner text, aria-label or aria-labelledby when needed.
  • Use aria-current, --selected variants and semantic tokens (semantic-info/semantic-success/semantic-warning/semantic-danger) to express state.
  • Use block shells (.card, .media, .list-group, .empty) for structured content only — let utilities handle spacing, grid and flow.
  • Use interactive variants (--interactive) only when the full block is intended to be actionable.
  • Use status tokens for semantic meaning: semantic-info, semantic-success, semantic-warning, semantic-danger.
  • Keep media blocks responsive by preserving structure: media__icon, media__img, media__body.
  • Use chips/tags/badges for short, high-signal labels; keep them concise.

Don’t

  • Don’t override block padding, radius, border or gaps per-instance — extend tokens, don’t break them.
  • Don’t turn every card or media block into a clickable element without clear intent or accessible labelling.
  • Don’t use content blocks to build page layout — they are layout-neutral shells.
  • Don’t remove focus-visible styles from interactive variants.
  • Don’t hard-code semantic colours for callouts, highlights, tags or chips — rely on theme-level semantic tokens.
  • Don’t encode navigation semantics inside non-interactive blocks (e.g., clickable <div> without proper role).
  • Don’t mix multiple interaction models inside a single block (e.g., nested --interactive regions). 
1. Use <button>/<a> for interactive cards, list items, chips and tags.
2. Provide accessible names for all interactive surfaces.
3. Apply semantic meaning via tokens (semantic-info/semantic-success/semantic-warning/semantic-danger) not custom colours.
4. Keep blocks layout-neutral; use utilities for spacing/grid/stack.
5. Preserve focus-visible for interactive variants (never remove).
6. Use --selected and aria-current to expose selection state.
7. Avoid per-instance overrides of spacing, borders, or radius.

Accessibility rules for marketing blocks

Marketing layouts must remain accessible and perceivable regardless of their visual expressiveness. These rules ensure information remains usable, navigable, and compliant across media grids, feature sections, Z-pattern layouts and CTA blocks.

Core rules

  • Provide descriptive alt text for all images used in feature grids, media blocks or testimonials.
  • Maintain a minimum contrast ratio according to theme tokens. Marketing layouts must not reduce contrast for stylistic effect.
  • Never use icon-only buttons in CTAs; pair icons with text labels.
  • Ensure tab order respects document flow — do not reorder elements visually without updating DOM structure.
  • Use semantic headings (<h2>, <h3>) to define sections, not styling classes.
  • Avoid conveying meaning solely through colour in feature blocks or hero metrics. Add text labels for clarity.
  • When using Z-pattern layouts, ensure keyboard and screen reader order match the logical flow of content.

Do

  • Use descriptive headings inside marketing features.
  • Label interactive elements clearly (“Learn more”, “Start now”).
  • Provide alt text for product/feature illustrations.
  • Keep paragraph line lengths readable (<65ch).

Don’t

  • Don’t rely on colour-only indicators in metrics or badges.
  • Don’t place text on images unless contrast is guaranteed.
  • Don’t insert decorative media into screen-reader flow (mark as aria-hidden="true" if needed).
  • Don’t use heading tags purely for styling hierarchy.

Content-block accessibility & semantics

Content-blocks support documentation and marketing layouts, but accessibility must remain first-class. These guidelines ensure blocks remain readable, navigable and semantically correct across all screen sizes.

Semantic structure

  • Use semantic headings (<h2>, <h3>) within .content-block__header.
  • Use paragraphs (<p>) and lists (<ul>) inside .content-block__body.
  • Mark decorative media as aria-hidden="true".
  • Always provide descriptive alt attributes for meaningful media.
  • Prefer semantic tags (<article>, <section>) when content-blocks represent standalone content.

Keyboard & interaction

  • Make the entire block interactive only when using .content-block--interactive.
  • Ensure focus-visible styles remain unobstructed by decoration.
  • Do not trap focus inside a content-block.
  • Footer buttons must be reachable in normal tab order.

Contrast & readability

  • Maintain sufficient contrast between text and background, including inset variants.
  • Keep paragraphs under 65ch for optimal readability.
  • Avoid placing text directly on complex imagery.
  • When using compact blocks, keep text size at var(--text-sm) or larger.

Do / Don’t for advanced marketing layouts

Advanced layouts like Z-patterns, media-left sections, feature grids and hero-with-media blocks require clear architectural discipline. These rules define how to design high-impact layouts without breaking structure, rhythm or semantics.

Do

  • Use grid-auto-fit-* for responsive marketing grids that collapse naturally on mobile.
  • Alternate media and body blocks in Z-pattern layouts for visual rhythm.
  • Use .content-block__media for centered, well-contained media.
  • Ensure CTA blocks have clear primary/secondary actions.
  • Keep spacing consistent using section stacks or grid gaps (never manual margin hacks).

Don’t

  • Don’t override padding or spacing inside content-blocks with manual margins — use density variants instead.
  • Don’t mix cards, callouts and content-blocks in the same grid unless intentional for communication hierarchy.
  • Don’t break grid flow by forcing fixed widths.
  • Don’t reorder content visually without matching DOM order.
  • Don’t pair unrelated media types (e.g., product photo + abstract pattern) in the same layout without intent.

Content-block Do / Don’t

Content-blocks are neutral containers for structured text, media and actions. They should remain visually consistent and lightweight. These rules ensure proper use across documentation, marketing and product layouts.

Do

  • Use .content-block--surface for primary blocks.
  • Use .content-block--inset for low-contrast nested areas.
  • Use .content-block--compact for dense UI contexts.
  • Use semantic tags inside blocks (headings, lists, paragraphs).
  • Wrap images in .content-block__media for centered containment.
  • Keep footer actions grouped via .content-block__footer.
  • Let grid utilities handle layout — no manual margin hacks.

Don’t

  • Don’t use content-blocks as replacements for cards or callouts.
  • Don’t mix padding utilities inside blocks — rely on variants.
  • Don’t position media outside .content-block__media.
  • Don’t insert unstructured HTML (loose divs with no semantic meaning).
  • Don’t use content-blocks as interactive surfaces unless using --interactive.
  • Don’t override border or shadow tokens with arbitrary values.
