Navigation

Primary navigation, sidebar, breadcrumbs, pagination and tabs — unified under a single spacing, typography and interaction system.

Source: navigation.css · Depends on base.css, typography.css, theme tokens in /themes/*.css.

Installation & dependencies

Navigation components depend on core layout, typography, icons and a theme layer. Load in this exact order.

  • base.css (reset + primitives)
  • layout.css (flex/grid + responsive tokens)
  • typography.css (type scale)
  • icons.css (mask icons)
  • navigation.css (this subsystem)
  • An active theme file (slate, navy, carbon)

Breadcrumbs

Linear page hierarchy with separators and current-page state support.

This version:

  • uses .breadcrumbs as the container
  • uses .breadcrumbs__item per node
  • uses .breadcrumbs__link for all but the current page
  • uses .breadcrumbs__separator between items
  • marks the current page with aria-current="page" (correct accessibility contract)
  • respects ellipsis, truncation, max-width constraints from your CSS
  • produces: Home / Projects / VDS Redesign

Toolbar

A flexible horizontal control bar used for page headers, headers, actions, filters, batch operations and contextual commands. Toolbars support density modifiers, title blocks, and left/right action groups.

Users

Structure

Toolbars provide a horizontal layout with smart spacing, allowing left and right sections to contain titles, filters, search inputs, tabs or buttons. They inherit full surface, border and radius tokens for use in dashboards and page headers.

  • .toolbar forms the container.
  • .toolbar__section groups content on the left side.
  • .toolbar__actions aligns interactive controls on the right.
  • .toolbar__title provides a semantic title area.
  • Density modifiers: .toolbar--dense, .toolbar--spacious.

Footer Navigation

A simplified, low-emphasis navigation variant used in footers and auxiliary surfaces. Footer navigation uses muted colors, vertical layout and reduced spacing to provide unobtrusive access to informational pages, policies and secondary actions.

About Privacy Terms

Structure

Footer navigation reuses the core .nav and .nav__link patterns but applies footer-specific color tokens and vertical alignment. The reduced emphasis helps keep primary navigation distinct while providing accessible paths to secondary pages.

  • .footer .nav converts navigation into a stacked vertical layout.
  • .footer .nav__link uses footer text colors and reduced opacity.
  • .nav--vertical aligns items to the left for consistent reading flow.
  • Active links gain full opacity and medium font-weight.

Default

FAQ Support Contact 
<footer class="footer">
    <nav class="nav">
        <a class="nav__link">FAQ</a>
        <a class="nav__link">Support</a>
        <a class="nav__link nav__link--active">Contact</a>
    </nav>
</footer>

Vertical Layout

Vertical navigation is the default form in footers, enabling compact grouping and clear scanability across legal and informational pages.

Company Blog Careers 
<footer class="footer">
    <nav class="nav nav--vertical">
        <a class="nav__link">Company</a>
        <a class="nav__link">Blog</a>
        <a class="nav__link nav__link--active">Careers</a>
    </nav>
</footer>

Behavior

  • Footer links use --color-footer-text at 85% opacity.
  • Hover removes opacity reduction for clarity.
  • Active state uses full opacity and font-weight: 500.
  • Horizontal footer nav is allowed but not recommended for dense menus.

Announcement

A full-width banner used for site-wide notices, version announcements, scheduled maintenance, product updates, legal information, or marketing messages. Announcements support icons, inline links, dismiss actions, and status variants.

Welcome to the new VDS 3.0 release. Learn more

Structure

Announcements are placed at the top of a layout and span the full viewport width. The inner container aligns content, icons, actions, and optional close controls.

  • .announcement creates the banner surface.
  • .announcement__inner constrains width and provides layout.
  • .announcement__content groups icon, text, and any inline links.
  • .announcement__icon allows any icon-mask usage.
  • .announcement__text contains the message.
  • .announcement__link styles inline links within the banner.
  • .announcement__close provides a dismiss action.
  • Status variants: .announcement--info, .announcement--success, .announcement--warning, .announcement--danger.

Navigation Brand & Actions

Brand and action areas define the left and right structural anchors inside top-level navigation bars. They group logos, product titles, buttons, account menus and other high-priority controls while maintaining alignment and spacing consistency.

VDS Vlah Design System

Structure

The brand area anchors the left side of the navigation bar and typically contains a logo and an optional product or section title. The actions area anchors the right side and holds important controls such as authentication buttons, account menus or theme toggles.

  • .nav__brand groups logo and name elements.
  • .nav__brand--compact reduces spacing for dense environments.
  • .nav__logo vertically centers any logo element.
  • .nav__actions positions button-like controls on the right.
  • Spacing is controlled exclusively by layout tokens.

Brand

VDS Dashboard 
<nav class="nav">
    <div class="nav__brand">
        <span class="nav__logo">
            <span class="logo">VDS</span>
        </span>

        <strong class="text-sm">Dashboard</strong>
    </div>
</nav>

Compact Brand

Use .nav__brand--compact in constrained environments or inside vertical navigation where spacing should be minimized.

V Docs 
<div class="nav__brand nav__brand--compact">
    <span class="nav__logo">
        <span class="logo">V</span>
    </span>
    <span class="text-sm">Docs</span>
</div>

Actions

The actions area aligns navigation controls on the right side of the bar. Typical contents include authentication actions, user menus, search buttons or theme toggles. 

<div class="nav__actions">
    <button class="button button--subtle">Help</button>
    <button class="button button--primary">Sign In</button>
</div>

Usage

  • Brand area always aligns to the left edge of the navigation container.
  • Actions appear right-aligned and follow interactive spacing rules.
  • Logos inside .nav__logo retain natural dimensions and vertical centering.
  • Product names, section identifiers or site titles may appear inside .nav__brand.
  • Do not overload .nav__actions with large forms; keep it interactive and lightweight.

Navigation Toggle

The navigation toggle provides a compact control for opening and closing mobile navigation menus. It supports hover, active and focus-visible states, as well as an .nav__toggle--active modifier for indicating the expanded menu state.

Structure

The toggle control is hidden by default in desktop navigation. At mobile breakpoints it becomes visible and triggers the display of the mobile menu or drawer. It uses tokens for radius, padding, transitions and focus-visible styling.

  • .nav__toggle is the interactive button.
  • SVG icon inherits color from its parent (currentColor).
  • Hover, active and focus-visible states follow button interaction rules.
  • .nav__toggle--active visually indicates an expanded or open menu.

Basic

<button class="nav__toggle">
    <svg viewBox="0 0 24 24" fill="none">
        <line x1="3" y1="6"  x2="21" y2="6"  stroke-width="2" />
        <line x1="3" y1="12" x2="21" y2="12" stroke-width="2" />
        <line x1="3" y1="18" x2="21" y2="18" stroke-width="2" />
    </svg>
</button>

Active State

Apply .nav__toggle--active when the mobile navigation is open or expanded. 

<button class="nav__toggle nav__toggle--active">
    <svg viewBox="0 0 24 24" fill="none">
        <line x1="3" y1="6"  x2="21" y2="6"  stroke-width="2" />
        <line x1="3" y1="12" x2="21" y2="12" stroke-width="2" />
        <line x1="3" y1="18" x2="21" y2="18" stroke-width="2" />
    </svg>
</button>

Usage

  • Toggle is hidden on desktop via media queries (see layout rules).
  • SVG uses stroke="currentColor" so the toggle can be themed via CSS.
  • Hover and active use subtle surface/background tokens for tactile feedback.
  • .nav__toggle--active maps directly to the expanded menu state.
  • Must include a focus-visible style for accessibility compliance.

Primary & Contextual Navigation

These navigation patterns provide horizontal menu structures for primary pages, contextual sections, and segmented choices. They include primary navigation, sub-navigation (tabs), and pill navigation for segmented filters or mode switching.

Home Projects Teams Billing

Primary Navigation

Primary navigation is used at the top-level of an application. It features padded, rounded items with subtle hover states and an accent background for active pages.

  • .nav--primary defines the horizontal layout.
  • Items use padding and rounded corners.
  • Hover triggers a subtle surface elevation.
  • Active state uses --color-accent-soft.
<nav class="nav nav--primary">
    <a class="nav__link nav__link--active">Home</a>
    <a class="nav__link">Projects</a>
    <a class="nav__link">Teams</a>
</nav>

Sub Navigation (Tabs)

Sub navigation is used for contextual page-level sections and behaves like a tab bar. It uses a bottom border with an accent indicator to mark the active section.

Overview Activity Integrations Settings
  • .nav--sub creates a tab-like bar.
  • Active tabs include an accent indicator via ::after.
  • Disabled items reduce opacity and remove interaction.
  • Full focus-visible ring is supported.
<nav class="nav nav--sub">
    <a class="nav__link nav__link--active">Overview</a>
    <a class="nav__link">Activity</a>
    <a class="nav__link">Integrations</a>
    <a class="nav__link nav__link--disabled">Settings</a>
</nav>

Pill Navigation

Pill navigation provides lightweight segmented controls used for filtering, mode selection, or compact alternative to tabs. Pills support wrapping and small sizes.

All Open Closed Archived
  • .nav--pills uses rounded pill shapes.
  • Bordered and surface-backed by default.
  • Active pills use accent border + soft accent background.
  • Disabled pills reduce opacity and remove pointer events.
<nav class="nav nav--pills">
    <a class="nav__link nav__link--active">All</a>
    <a class="nav__link">Open</a>
    <a class="nav__link">Closed</a>
</nav>

Behavior

  • Primary navigation is typically placed in top-level app bars.
  • Sub navigation (tabs) is used for contextual sections within a page.
  • Pill navigation is flexible and works across toolbars, cards and filters.
  • All navigation items support :hover and :focus-visible states.
  • Spacing and radii are token-driven and theme-consistent.

Accessibility & usage rules

Navigation components must preserve semantic roles, keyboard-first operability, predictable ARIA states, and token-driven contrast. Horizontal nav, vertical nav, breadcrumbs, pagination, tabs and mobile panels all follow a single accessibility contract.

Do

  • Use the correct semantic container: <nav> for navigation regions, <ul>/<li> if using semantics, <button> for pagination and tab triggers.
  • Mark the current page explicitly with aria-current="page" on .nav__item--active, .sidebar__nav .nav__item--active, .pagination__item--active, aria-current="page" on the active .breadcrumbs__link.
  • Keep :focus-visible intact — navigation relies on token-driven focus rings for full keyboard operability.
  • Provide accessible labels on icon-only items inside .nav__item or .nav-tabs__item.
  • In mobile contexts, ensure: .nav__toggle includes aria-controls and aria-expanded, updated by the controlling script.
  • For tabs: role="tablist" on the container, role="tab" on each item, and keyboard behaviour (Left/Right) if ARIA tabs are fully implemented.
  • Use semantic separators in breadcrumbs and maintain linear order.
  • Apply disabled state semantically: disabled + .pagination__item--disabled or .nav__item--disabled.
  • Keep mobile panels (.nav-panel) paired with a backdrop (.nav-backdrop) to ensure proper focus trapping and dismiss behaviour.
  • Use .nav--vertical, .sidebar and depth utilities (--depth-2, --depth-3) to expose hierarchy visually without altering DOM reading order.

Don’t

  • Don’t simulate links or triggers with <div> elements — use <a> for navigation, <button> for actions.
  • Don’t hide focus-visible or replace it with non-token shadows.
  • Don’t remove breadcrumb separators; they provide navigational clarity and assistive-tech parsing.
  • Don’t mix navigation patterns (primary/tabs/pills/vertical) inside the same container. Use one pattern per context.
  • Don’t reorder DOM items to produce visual hierarchy — use depth classes.
  • Don’t override spacing, typography or radius with inline CSS; navigation must remain token-driven for consistency.
  • Don’t expose mobile panels (.nav-panel) without controlling aria-expanded or without showing .nav-backdrop.
  • Don’t apply multiple nav-level modifiers simultaneously (.nav--primary + .nav--pills, etc.).
  • Don’t use disabled links (<a disabled>) — invalid HTML. Use <button disabled> or style with .nav__item--disabled. 
1. Use <nav> semantics; links = <a>, triggers = <button>.
2. Mark current page with aria-current="page".
3. Never remove :focus-visible; token rings are mandatory.
4. Tabs: role="tablist", each tab = role="tab".
5. Mobile toggles: aria-controls + aria-expanded (JS-updated).
6. Disabled state: disabled attribute or --disabled modifier.
7. Don’t reorder DOM; use depth utilities for visual hierarchy.
8. Mobile panel requires backdrop for dismissal & focus safety.
9. One navigation pattern per surface (primary/pills/tabs/vertical).
10. Breadcrumbs must retain separators + current-page marking.
