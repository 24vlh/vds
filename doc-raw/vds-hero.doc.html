Hero

Primary page-intro system: landing hero, overview hero, KPI hero, media hero, loading and error hero — all token-driven and theme-bound.

Source: hero.css · Depends on primitives.css, base.css, layout.css, typography.css, utilities and theme tokens in /css/themes/*.css.

Installation & dependencies

Hero relies on core layout, typography and theme tokens. Metrics, media and actions reuse buttons, icons and generic utilities.

  • primitives.css — design tokens (spacing, radius, z-index, etc.).
  • base.css — reset, base element styles.
  • layout.css — content width and page containers.
  • typography.css — type scale and text utilities.
  • utilities.css — small utility helpers for spacing, text, etc.
  • buttons.css — CTA buttons used in hero actions.
  • icons.css — inline icons for hero points and metrics.
  • sections.css — page sections and inner containers.
  • hero.css — this hero subsystem (all variants).
  • One theme in /css/themes/*.css for colours and surfaces.

Base structure

Base hero is a padded band with an inner container, textual body and actions. All other variants extend this structure.

Product

Base hero headline

Short supporting description. Keep this to 1–2 lines for most layouts. 

<section class="hero hero--b">
  <div class="hero__inner">
    <div class="hero__body">
      <div class="hero__eyebrow">Section label</div>
      <h1 class="hero__title">Hero headline</h1>
      <p class="hero__subtitle">
        Supporting copy. 1–2 lines describing what lives behind this surface.
      </p>
      <div class="hero__actions">
        <button class="button button--primary">Primary</button>
        <button class="button button--ghost">Secondary</button>
      </div>
    </div>
  </div>
</section>

Responsive behaviour

Layout adjusts at tablet and mobile breakpoints: split/media heroes collapse to a single column, metrics grid reduces columns, and padding scales down.

Behaviour is fully controlled by @media rules in hero.css. No additional markup is required.

  • .hero--split .hero__inner → grid collapses to 1 column.
  • .hero--with-media .hero__inner → grid collapses to 1 column.
  • .hero__metrics → auto-fit grid, fewer columns on small screens.
  • Padding scales down for .hero--a/b/c at smaller viewports.
/* hero.css (excerpt – actual values use px breakpoints) */
@media (max-width: 1024px) {
  .hero--split .hero__inner,
  .hero--with-media .hero__inner {
    grid-template-columns: 1fr;
  }
}

@media (max-width: 768px) {
  .hero {
    padding-block: var(--space-10) var(--space-14);
  }
  .hero__title {
    font-size: var(--text-3xl);
  }
}

@media (max-width: 640px) {
  .hero__title {
    font-size: var(--text-2xl);
  }

  .hero__metrics {
    grid-template-columns: repeat(
      auto-fit,
      minmax(calc(var(--hero-metric-min-width) * 0.8), 1fr)
    );
  }
}

Accessibility & usage rules

Hero surfaces are high-visibility entry points. They must preserve semantic heading hierarchy, expose readable metrics, guarantee contrast across variants and maintain predictable keyboard and screen-reader behaviour across all themes and densities.

Do

  • Use a real heading structure: <h1> for the primary hero title, <h2>/<h3> for subordinate sections below it.
  • Provide clear, unambiguous button labels in .hero__actions; labels must communicate outcome (“View metrics”, not “Click”).
  • Use .hero__eyebrow as a semantic category label (not decorative text). It is read by screen readers and styled via tokenised uppercase + letter-spacing.
  • Ensure adequate contrast in .hero--full: colours derive from theme tokens and overlay handling must maintain AA contrast.
  • In split / media heroes (.hero--split, .hero--with-media), provide alt text for images or mark decorative media with alt="".
  • For KPI surfaces (.hero__metrics), always pair: .hero__metric-label + .hero__metric-value. Trend indicators (--up, --down, --neutral) must use tokenised success/danger/soft colours and clear text.
  • Keep loading state semantic: .hero--loading hides .hero__body visually and reveals a token-driven skeleton shimmer — assistive tech still reaches the underlying structure.
  • Preserve spacing, radius and surface tokens; all variants (A/B/C/compact/minimal) inherit padding and text scale from design tokens.
  • Ensure error state contains a clear .hero__error-title and actionable .hero__error-actions (retry, view incident, etc.).

Don’t

  • Don’t demote or promote heading levels to achieve visual size — typography tokens control scale; headings control semantics.
  • Don’t place more than two primary CTAs in .hero__actions; excess actions break visual hierarchy and overwhelm screen-reader users.
  • Don’t combine mutually exclusive layout variants (.hero--split, .hero--with-media, .hero--full, .hero--inset) on the same hero.
  • Don’t rely solely on colour to communicate metric state — trend labels must include text (“+8.2% up”, “–0.3% down”, “unchanged”).
  • Don’t override token-driven contrast, radius or shadows in .hero--full or .hero--inset.
  • Don’t hide critical hero content behind decorative media or overlays.
  • Don’t interfere with skeleton animation timing (--hero-skeleton-speed) or keyframes — loading state is part of the UX pattern. 
1. One h1 in the hero; all secondary headings demoted properly.
2. .hero__actions must use real <button> / <a> labels with clear verbs.
3. Metrics: always label (metric-label) + value (metric-value) + trend text.
4. Media: provide meaningful alt text or mark as decorative (alt="").
5. Full heroes must maintain token-driven contrast and overlay readability.
6. Layout variants are mutually exclusive; do not combine them.
7. Loading state: hide .hero__body visually but keep structure for AT.
8. Error state: provide clear message + remediation path via .hero__error-title and .hero__error-actions.
9. Never override spacing, colour or radius tokens in hero markup.
