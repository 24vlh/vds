Tooltips & Popovers

Token-driven micro-surfaces for contextual help, hints and interactive overlays anchored to triggers without disrupting layout.

Installation & dependencies

Tooltips and popovers are surface-level components depending on base reset, layout, typography, tokens and the active theme. Include files in this order for predictable rendering and theming.:contentReference[oaicite:1]{index=1}

  • Load base.css and layout.css.
  • Load typography.css and icons.css.
  • Load active theme: themes/slate.css (or navy.css, carbon.css), which provides semantic surfaces and feedback tokens.:contentReference[oaicite:2]{index=2}
  • Load toasts-popovers.css for tooltips and popovers.:contentReference[oaicite:3]{index=3}
  • Load doc-block.css only for documentation pages.:contentReference[oaicite:4]{index=4}

Anatomy: tooltip & popover

The tooltip is a non-interactive micro-surface for short contextual help. The popover is an interactive micro-surface holding menus, forms or controls, anchored to a trigger without blocking the page.:contentReference[oaicite:5]{index=5}

Tooltip: requires a positioned ancestor (.tooltip-trigger) and one of the position modifiers (tooltip--top|bottom|left|right).:contentReference[oaicite:6]{index=6}
Popover: uses .popover-anchor as positioning context and accepts placement modifiers such as popover--bottom-right.:contentReference[oaicite:7]{index=7}

Popover menus & filter panels

Popovers can render menus, compact forms or filter panels without invoking a full modal or drawer. Position and density are controlled by class modifiers; behaviour is owned by your JavaScript.:contentReference[oaicite:11]{index=11}

Accessibility & usage rules

Contract for safe, semantic, token-driven tooltips and popovers. Maintain correct ARIA roles, connect triggers to surfaces, and rely on VDS tokens for motion, surfaces and focus instead of ad-hoc styling.:contentReference[oaicite:12]{index=12}

Do

  • Use role="tooltip" with aria-describedby on the trigger to connect tooltips to their anchors.
  • Use role="dialog" or role="menu" for popovers, depending on whether they behave as dialogs or menus.
  • Manage open/close state via JavaScript by toggling data-state="open" or .tooltip--visible / .popover--open.:contentReference[oaicite:13]{index=13}
  • Keep layout, shadow and colour decisions inside toasts-popovers.css using tokens for radii, borders, shadows and motion.:contentReference[oaicite:14]{index=14}
  • Prefer concise, task-focused copy in tooltips; use popovers when the content becomes actionable (menus, forms, controls).
  • Ensure triggers are keyboard focusable (buttons, links, or elements with tabindex="0" and correct roles).
  • Use semantic variants (--info, --success, --warning, --danger) to express meaning consistently with other feedback components.:contentReference[oaicite:15]{index=15}

Don’t

  • Don’t place actionable controls (buttons, links, inputs) inside a tooltip. Use a popover for interactive content.
  • Don’t hide critical information only in tooltips; treat tooltips as enhancements, not the sole source of requirements or warnings.
  • Don’t suppress focus-visible outlines on triggers or rely exclusively on hover to reveal tooltips; keyboard users must be able to access the same information.
  • Don’t animate tooltips or popovers with custom transitions that differ from token-driven motion. Use --transition-fast and --ease-out from primitives.:contentReference[oaicite:16]{index=16}
  • Don’t mix multiple independent popovers open at once in dense UIs unless there is a clear, intentional UX reason.
  • Don’t override popover width with arbitrary inline styles; rely on the min/max constraints in the component and design content accordingly.:contentReference[oaicite:17]{index=17} 
1. Use role="tooltip" with aria-describedby for tooltips.
2. Use role="dialog" or role="menu" for popovers + aria-haspopup/expanded/controls.
3. Toggle data-state="open" (or .tooltip--visible / .popover--open) from JS; CSS never owns behaviour.
4. Keep interactive controls out of tooltips; use popovers for menus/forms.
5. Keep motion, radius, shadow and colours token-driven inside toasts-popovers.css.
6. Preserve focus-visible outlines and keyboard reachability for all triggers.
7. Use semantic variants consistently with other VDS feedback components.
