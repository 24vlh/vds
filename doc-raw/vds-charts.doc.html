Charts & data visualisation

Token-driven chart shells for dashboards, reports and application metrics. Charts provide containers, axes, legends and state handling for any SVG/canvas-based library without locking you into a specific JS implementation.

Source: charts.css · Sits above primitives.css, themes/*.css, base.css, layout.css, typography.css and content-blocks.css.

Installation & dependencies

Charts sit on top of the core VDS stack and reuse existing tokens for spacing, surfaces, typography and motion. They are theme-bound through the active identity file and neutral primitives.

  • Load neutral tokens: primitives.css.
  • Load one active theme: themes/slate.css, themes/navy.css or themes/carbon.css.
  • Load core layers: base.css, layout.css, typography.css, utilities.css.
  • Load supporting layers as needed: buttons.css, icons.css, content-blocks.css, tables.css.
  • Load charts.css for chart shells and data visualisation tokens.
  • Load doc-block.css for documentation pages only (not required in production).

Accessibility & usage rules

Contract for safe, semantic composition of chart shells, legends, axes and states. Charts visualise data but do not own navigation or layout; assistive technology must always have an equivalent textual path to the same information.

Do

  • Expose a textual summary for each chart using nearby body copy or aria-label / aria-describedby on the canvas/SVG container.
  • Use role="img" and an accessible name for simple charts, or mark them aria-hidden="true" when a table or stat block already exposes the same information.
  • Keep charts layout-neutral: use .section, grids and cards for layout and density, not custom chart margins.
  • Use chart tokens (series, semantic positive/negative) instead of hard-coded colours for series, gridlines and axes.
  • Provide explicit empty, error and loading states using .chart--empty, .chart--error, .chart--loading and .chart-state.
  • Use legends for multi-series charts and keep labels short but meaningful; map labels to business terms users recognise.
  • Align chart time ranges and metrics with surrounding stats and tables to avoid misinterpretation (e.g. same period, same segmentation).

Don’t

  • Don’t rely on colour alone to communicate critical meaning (good vs bad); combine colour with shape, labels or iconography.
  • Don’t use charts as the only source of truth for important numbers; mirror key values in text, stats or tables.
  • Don’t place interactive controls inside the canvas/SVG region without proper keyboard support and focus management.
  • Don’t override chart padding, radius or borders per-instance; extend tokens globally if you need new densities.
  • Don’t hijack chart titles or legends for navigation; use proper navigation components for filtering and routing.
  • Don’t mix multiple interaction models in one chart (e.g. drag-to-zoom plus click-to-select plus hover tooltips) without clear affordances.
  • Don’t turn charts into layout containers; never nest unrelated content or form controls inside .chart__canvas. 
1. Provide a textual equivalent for all critical data shown in charts.
2. Use chart shells (.chart, .chart__header/body/legend) as visual containers only.
3. Style axes, gridlines and series via semantic classes, not ad-hoc colours.
4. Use explicit empty/error/loading states instead of leaving blank canvases.
5. Keep charts layout-neutral; rely on sections, grids and cards for page structure.
6. Avoid chart-only navigation; keep routing and filters in dedicated components.
7. Prefer tables or stats when exact values matter more than shape or trend.
