Toast System

The VDS Toast Layer provides ephemeral, system-level notifications: floating stacks, inline toasts, semantic variants and auto-dismiss timers. Toasts complement alerts and banners by confirming actions, surfacing background events and reporting non-blocking errors without disrupting flow.

Source: toasts.css · Sits above primitives.css, themes/slate.css, base.css, layout.css, utilities.css, doc-block.css and feedback.css.

This page documents all primitives exported by toasts.css. Toasts come in floating and inline variants, support full semantic variants (neutral, info, success, warning, danger/error), and are designed for keyboard-accessible, motion-aware, token-driven behaviour.

Overview & Dependencies

toasts.css is a component-layer extension of the Feedback System. It defines toast stacks, base toast anatomy, semantic variants, inline mode and motion hooks. All spacing, colors, border-radius, shadows and transitions are provided by core VDS tokens.

  • Floating stacks use .toast-stack with positional modifiers such as .toast-stack--top-right.
  • Each toast uses a consistent structure: .toast, .toast__icon, .toast__body, .toast__title, .toast__message, .toast__close and .toast__progress.
  • Semantic variants are mapped with modifiers: .toast--neutral, .toast--info, .toast--success, .toast--warning, .toast--danger.
  • Auto-dismiss timers are configured via the --toast-duration custom property.
  • Animations respect prefers-reduced-motion and fall back to non-animated appearance.
  • Toasts rely on the same focus-visible and semantic color contracts as alerts, banners and other feedback primitives.

Floating toasts are rendered inside fixed-position stacks. Each stack controls placement (top/bottom, left/right/center) and vertical spacing. On small screens, stacks collapse into a full-width bottom position for better reachability.

Base Toast Anatomy

The base toast structure is layout-stable: slots for icon, content, close button and optional progress bar. Grid areas keep the icon, text and close control aligned while preserving small hit targets and succinct content.

Profile saved
Your profile settings have been successfully updated.
  • Use role="status" for non-blocking updates.
  • Reserve role="alert" for critical errors that require immediate attention.
  • Always provide an accessible label for the dismiss button.

     aria-live="polite"

Inline vs Floating Toasts

The same component can be used as a floating toast inside a stack or as an inline block within a card or section. Inline toasts remove elevation and progress bar, aligning visually with surrounding content.

Billing Settings

Changes to your billing details may take a few minutes to propagate.

Heads up
Your plan will renew on 1st of next month.

Inline toasts sit inside the layout flow and use the same semantic variants as floating toasts, but with no elevation or progress bar. 

<div class="toast-stack toast-stack--bottom-right">
  <div class="toast toast--success">...</div>
</div>

<div class="card">
  <div class="card__body">

    <div class="toast toast--inline toast--info">

Auto-Dismiss & Duration

Toasts expose a --toast-duration property used by the progress bar. JavaScript can read the same value to schedule auto-dismiss. Keep durations short, predictable and token-consistent.

Checklist completed.
Connection unstable. Retrying…

Use shorter durations for success/info, slightly longer for warnings. Critical errors should remain visible until dismissed. 

<div class="toast toast--success">...</div>

<div class="toast toast--success"
     style="--toast-duration: 4000ms;">
  ...
</div>

<div class="toast toast--warning"
     style="--toast-duration: 9000ms;">
  ...
</div>

Motion & Reduced Motion

Toast animations use token-based easing and durations, with automatic fallbacks for users who prefer reduced motion. Enter and exit transitions are subtle by default and must not be overridden with aggressive effects.

  • Entry uses a short slide-and-fade animation based on var(--transition-normal) and var(--ease-out).
  • Exit uses a slightly faster fade-out with var(--transition-fast) and var(--ease-in).
  • prefers-reduced-motion: reduce disables animations and relies on opacity only.
  • Do not chain additional JS-based animations on top of the core motion; instead, adjust tokens if needed.
  • Progress bar animation uses the same duration as --toast-duration to stay in sync with auto-dismiss. 
@keyframes toast-slide-in {
  from { opacity: 0; transform: translateY(8px); }
  to   { opacity: 1; transform: translateY(0); }
}

@keyframes toast-slide-out {
  from { opacity: 1; transform: translateY(0); }
  to   { opacity: 0; transform: translateY(6px); }
}

@keyframes toast-progress {
  from { transform: scaleX(1); }
  to   { transform: scaleX(0); }
}

@media (prefers-reduced-motion: reduce) {
  .toast,
  .toast--exiting {
    animation: none;
    transition: none;
  }

  .toast__progress::before {
    animation-duration: var(--motion-none);
  }
}

Accessibility & Usage Rules

Contract for accessible, semantic, token-driven toast notifications. Ensure toasts remain concise, dismissible, keyboard-accessible and correctly scoped to their semantic meaning and urgency.

Do

  • Use semantic modifiers (.toast--info, .toast--success, .toast--warning, .toast--danger, .toast--neutral) to convey real meaning.
  • Set role="status" and aria-live="polite" for non-critical updates, reserving role="alert" for high-severity errors.
  • Always provide accessible labels for dismiss buttons: aria-label="Dismiss notification".
  • Keep toast content short: one title or one sentence. Use dialogs or banners for complex messages.
  • Ensure all toast dismissal controls are reachable via keyboard and expose visible focus states.
  • Use inline toasts for contextual notices within cards; use floating stacks for global or cross-context events.
  • Respect user motion preferences; rely on existing token-based animations rather than hard-coded custom effects.

Don’t

  • Don’t overload toasts with long paragraphs, lists or complex forms—this breaks the ephemeral contract.
  • Don’t use .toast--danger for low-risk or informational events; reserve it for real problems.
  • Don’t remove or override the dismiss button; every toast must be dismissible by the user.
  • Don’t use multiple conflicting semantic variants on the same toast or mix conflicting messages in a single notification.
  • Don’t trigger rapid sequences of toasts; batch or debounce where possible to avoid overwhelming the user.
  • Don’t animate toasts with large translations, bounces or spins; keep motion subtle and token-consistent.
  • Don’t hijack toast areas for persistent banners; use banners or alerts for long-lived feedback instead. 
1. Use semantic modifiers (.toast--info/success/warning/danger/neutral) for real meaning.
2. Use role="status" + aria-live="polite" for non-critical events; role="alert" only for failures.
3. Provide aria-labels for all dismiss buttons and keep them keyboard accessible.
4. Keep toasts short-lived, concise and dismissible; use banners/dialogs for complex content.
5. Configure durations via --toast-duration and align JS auto-dismiss with the same value.
6. Respect prefers-reduced-motion; never add aggressive or custom motion on top.
7. Avoid flooding users with toasts; queue, debounce or aggregate events as needed.
8. Use inline toasts for local context; use floating stacks for global or cross-context status.
