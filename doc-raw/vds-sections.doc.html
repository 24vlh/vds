Layout surfaces

Sections System (V2)

Structural building blocks for page-level content. V2 keeps legacy densities (A/B/C) and adds universal rhythm (XS–XL), inner width + edge controls, per-edge padding, clean split reversal, inset variants, grid gaps, stacks, inline stacks, and value lists — all token-driven.

Backwards compatible
Existing pages using A/B/C, split, grids, stacks and lists continue to work.
Universal rhythm
New .section--xs/sm/md/lg/xl uses primitives section-spacing tokens for predictable pacing.
Composable layouts
Split sections, grids, stacks and lists compose into landing pages, docs, dashboards, and utilities.

Installation & dependencies

Sections are structural only. They depend on system tokens and typography. They do not ship standalone components — they host components.

  • primitives.css — global VDS tokens (spacing, radii, layout widths).
  • base.css — resets and semantic defaults.
  • typography.css — headings/body scale used in section headers.
  • sections.css — this subsystem (V2).
  • doc-block.css — documentation engine (for this page only).
  • content-blocks.css — cards, features etc. used in examples.
  • One active theme (slate.css / others) for colours.

Responsive behaviour

Sections collapse split layouts and reflow grids at system breakpoints. No markup changes needed.

  • .section--split and .section--split-reverse collapse to one column below 1024px.
  • All grids auto-fit to smaller min-widths at 1280 / 1024 / 768 breakpoints.
  • Legacy densities shrink vertical padding to --space-10 on small screens.
  • Inset padding reduces on mobile for readability.
/* sections.css (excerpt from V2) */
@media (max-width: 1024px) {
    .section--split .section__inner,
    .section--split-reverse .section__inner {
        grid-template-columns: 1fr;
    }
}

@media (max-width: 768px) {
    .section,
    .section--a,
    .section--b,
    .section--c {
        padding-top: var(--space-10);
        padding-bottom: var(--space-10);
    }
}

Usage

  • Use .section--surface for card-like or interactive blocks.
  • Use .section--subtle to group related content lightly.
  • Use .section--brand for marketing tiers or highlighted panels.
  • Alternate backgrounds help divide long pages without borders.
  • Split layouts should be used for text + image or dual-column content.
  • Reversed split should only be used for layout alternation, not content dependence.

Accessibility & usage rules

Sections provide structural rhythm, containment, spacing and responsive flow. They are visually expressive but intentionally non-semantic. Accessibility comes from correct HTML landmarks, heading hierarchy, predictable reading order and token-driven contrast — never from the classes themselves.

Do

  • Use real HTML landmarks: <section>, <article>, <aside>, <main>.
    Sections.css does not provide semantics.
  • Use aria-labelledby when a section requires a programmatic name, pairing it with the corresponding .section__title.
  • Maintain strict heading hierarchy: <h1> only once per page, then <h2> for each major section, <h3> inside subsections.
  • Choose density (A/B/C or XS–XL) intentionally — these modify only vertical padding and do not affect structure or semantics.
  • Use .section__inner to enforce readable line length (--layout-max-width) and proper horizontal padding.
  • For split layouts (.section--split / .section--split-reverse), keep the DOM order meaningful. CSS reverses columns visually; assistive tech follows DOM.
  • Ensure icon-only list items inside .section__list still expose text labels — icons are decorative.
  • Maintain token-derived spacing: stacks (.section__stack*), insets, and grids rely on --space-* and --section-spacing-*.
  • Verify color contrast for surface/subtle/brand/alt backgrounds. These classes apply backgrounds but do not override text logic.
  • Treat inset surfaces (.section--inset, --inset-sm, --inset-lg) as visual containers only — semantics stay inside.

Don’t

  • Don’t treat .section, .section--surface or .section--subtle as semantic roles — they do not replace <section>, <article>, etc.
  • Don’t reorder heading hierarchy to achieve visual size; typography tokens handle scale, not incorrect heading levels.
  • Don’t use densitiy modifiers (--a, --b, --c, --xs…--xl) to force semantic grouping — they are spacing only.
  • Don’t override spacing or padding inside .section directly. Always extend via tokens or use stacks / insets.
  • Don’t fake split reversal via DOM manipulation — V2 provides clean reversal (.section--split-reverse) without RTL hacks.
  • Don’t place full-width components inside .section--inset if they already provide their own outer padding — use .section--flush.
  • Don’t set fixed heights or widths on grid children — section grids are responsive and token-driven.
  • Don’t apply interactive behaviour (toggles, overlays, drawers) to section classes. Sections are structural only.
  • Don’t mix too many rhythm systems simultaneously (A/B/C + XS–XL + custom margins). Choose one vertical pacing logic for clarity. 
1. .section is visual; semantics come from the HTML tag.
2. Use aria-labelledby to name sections programmatically.
3. Maintain heading hierarchy (H1 once; H2 per major band).
4. Use .section__inner for readable line length.
5. Split: DOM order = reading order; CSS handles visual swap.
6. Density (A/B/C or XS–XL) controls spacing only.
7. Surface/subtle/brand backgrounds require contrast checks.
8. Insets provide visual containment, not semantics.
9. Grids/stacks are layout only; no role or ARIA implied.
10. Never override padding/margins inline; extend tokens instead.
