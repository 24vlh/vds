Skeleton loading system

Token-driven skeleton surfaces for perceived-performance: lines, avatars, blocks and cards that mirror real layouts while content is loading. Designed for dashboards, SaaS, forms, tables and marketing surfaces.

Source: skeleton.css · Depends on primitives.css, themes/*.css, base.css, layout.css, typography.css, utilities.css, content-blocks.css, forms.css and tables.css.

Installation & dependencies

skeleton.css is a thin presentation layer that reuses existing layout, content-block and form structures. Load it after core layout and components so it can target shared classes safely.

  • Load neutral tokens: primitives.css.
  • Load one active theme from /themes/*.css.
  • Load core structural layers: base.css, layout.css, typography.css.
  • Load primary components: content-blocks.css, forms.css, tables.css, buttons.css (optional).
  • Load utilities: utilities.css for flex, gap, grid and spacing.
  • Then load skeleton.css to enable skeleton states for cards, forms, lists and tables.
  • For documentation pages, also load doc-block.css.

Core concept & goals

Skeletons are layout-accurate placeholders that represent the structure of content while data is loading. They reduce perceived latency, avoid layout shift, and keep focus on the UI rather than on spinners.

Use skeletons when content will appear in the same place (same card, same row, same column). Use spinners only for background tasks that do not change layout.

  • Structure-first — mirror the real layout: avatar, title, meta, actions, body.
  • Token-driven — colours, radii, gaps and animation are sourced from theme and primitives.
  • Compositional — skeletons slot into cards, forms, tables and grids without dedicated wrappers.
  • Accessible — motion is optional, ARIA attributes keep assistive tech informed, contrast comes from surface tokens.

Anatomy & API overview

The skeleton system is built from a small vocabulary of primitives and modifiers. Each class has a clear structural role; variants are additive and token-driven.

Core primitives

.skeleton
Base surface; shared animation, colour and radius.
.skeleton-line
Horizontal bar for titles, text and metadata.
.skeleton-avatar
Circle or square avatar placeholder.
.skeleton-block
Generic rectangular block (media, charts, thumbnails).
.skeleton-card
Composite card layout mirroring real card content.

Modifiers

--sm / --md / --lg
Size modifiers for height / diameter.
.skeleton-line--short, --medium, --long
Width presets for textual lines.
.skeleton--subtle, .skeleton--strong
Colour intensity variants.
.skeleton--static
Disables animation for reduced-motion users or cheap surfaces.

Variants, density & motion

Skeleton variants handle size, density and intensity without custom colours or inline spacing. Motion can be disabled globally or per element.

Size modifiers align skeleton rhythm with form densities and card scales. Avoid arbitrary pixel overrides; prefer class-based presets.

Accessibility & usage rules

Contract for accessible, predictable skeleton usage across VDS. Skeletons must respect user motion preferences, preserve focus order, and never hide critical state information.

Do

  • Use skeletons only while content is genuinely loading. As soon as data is available, replace the skeleton markup instead of layering content on top.
  • Mirror real layouts: if the final card has avatar, title, meta and chart, the skeleton should have avatar + lines + block in the same positions.
  • Respect prefers-reduced-motion by disabling shimmer globally or via .skeleton--static.
  • Keep focusable elements out of skeleton surfaces. While loading, disable or remove interactive controls rather than letting users tab into emptiness.
  • Use semantic wrappers (cards, form-field, table rows) and place skeletons inside them — this keeps the DOM stable and avoids layout jumps.
  • Use token-driven colours and radii only; never hard-code skeleton palettes per page.
  • Communicate loading state to assistive tech via container-level attributes (e.g. aria-busy="true" on the card or region).

Don’t

  • Don’t leave skeletons visible after content has loaded — this creates visual noise and suggests something is still loading.
  • Don’t animate large surfaces aggressively. Prefer subtle, low-contrast motion or static skeletons, especially on dashboards with many items.
  • Don’t change layout between skeleton and loaded states (different number of cards, different column count, etc.).
  • Don’t override skeleton colours with arbitrary brand colours. Use theme tokens so light/dark modes remain consistent.
  • Don’t hide critical error or empty states behind skeletons. If a request fails, switch to a clear error or retry surface instead of leaving the skeleton running.
  • Don’t use skeletons for instantaneous operations (sub-100ms). In those cases, render the final UI without transition.

            <div class="skeleton skeleton-block" style="width: 100%; height: 160px;"></div>
        </div>
    </div>
