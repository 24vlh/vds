Command Palette

Command Menu / Spotlight — Global Action Surface

A global, keyboard-first command surface for navigation, actions and search. The command palette sits above the rest of the UI as a modal overlay, combining a search input, grouped result lists, keyboard shortcuts and helper text.

Source: command.css · Depends on primitives.css, base.css, layout.css, utilities.css, typography.css, active theme tokens (e.g. slate.css), and optional icons.css, overlays.css.

Installation & dependencies

Load command.css after base, layout and utilities. Use data-target="key" on buttons, pair it with .command panels carrying data-command="key", and toggle command--open on the root element to control visibility.

CSS load order

  • 1. Tokens & base primitives.css, base.css, theme file (e.g. slate.css)
  • 2. Layout & utilities layout.css, utilities.css, typography.css
  • 3. Optional overlays overlays.css for shared z-index and overlay behaviour
  • 4. Command palette command.css for panel, list and item styling

JavaScript contract (canonical)

Command palette visibility is controlled via: data-target="key" on triggers, .command[data-command="key"] on panels, and .command--open on the active panel.

Contract: every button declares data-target="basic". The matching panel declares the same data-command="basic" value on the root .command. Open state is represented by .command--open.

Concept

What the Command Palette Is

The Command Palette is the central interaction surface of a modern application. It unifies navigation, action execution, search, shortcuts and context-switching into a single, keyboard-first interface that can be opened from anywhere within the product.

1. A Universal Interface for the Entire Application

A Command Palette is a single panel—typically opened with ⌘K or CtrlK—that allows users to:

  • Navigate anywhere in the product
  • Run actions and workflows
  • Open tools, settings or system panels
  • Search entities and content
  • Switch workspaces, accounts or contexts
  • Discover keyboard shortcuts

It compresses the entire system’s capability into a single point of access. Instead of learning complex navigation trees, the user learns one shortcut and one surface.

2. Why Modern SaaS Platforms Use It

As products grow, traditional navigation becomes slow and fragmented. A Command Palette solves four structural problems:

  • Speed: reach anything in <1 second.
  • Discoverability: reveal full product capability through search.
  • Consistency: one model—no relearning UI layouts.
  • Keyboard mastery: enables high-performance workflows.

This is why systems like Linear, Raycast, VSCode, GitHub Spotlight and Notion treat the palette as a core feature, not an optional enhancement.

3. How the User Interacts With It

1. Shortcut
The palette opens and focuses the search input.

2. Type intent
“billing”, “create project”, “invite”, “dark mode”.

3. Navigate
Use ↑ ↓ to move results, Enter to run, Tab to switch groups, Esc to close.

4. Execute
The app navigates, opens, switches or performs the action.

4. What the Command Palette Shows

  • Search input with live filtering
  • Recent commands or navigation targets
  • Grouped command categories
  • Keyboard shortcuts and hints
  • Workspace or account switchers
  • Empty state for unmatched queries
  • Footer with navigation hints

This turns the palette into a unified search + navigation + action surface.

5. Why It Is Not Just a Search Box

While it resembles a search bar or modal, its purpose is broader. A Command Palette is not:

  • a dropdown
  • a simple search bar
  • a menu
  • a modal dialog

It is the orchestration surface of the product—where navigation, actions, search and shortcuts converge.

6. The Core Mental Model

The palette acts as a universal interface between intention and capability. Users express intent such as:

  • “Go to billing”
  • “Create project”
  • “Switch workspace”
  • “Invite teammate”

The palette interprets that intent and routes the system to the correct action.

7. Core Responsibilities of the Palette

  • Global navigation
  • Action execution
  • Search and lookup
  • Context switching
  • Shortcut discovery
  • Zero-state guidance

8. Why This Matters for VDS

The Command Palette is a first-class component in VDS because it offers:

  • A single architectural point for commands
  • Horizontal scalability as features grow
  • A premium interaction pattern expected by modern users
  • High-speed, keyboard-driven workflows
  • Centralised discoverability

9. Where It Lives in the UI

The palette always lives:

  • globally, accessible from any page
  • above all other UI layers
  • keyboard accessible
  • dismissible via Esc
  • vertically centered by default
  • triggerable from buttons or shortcuts

It is the fastest path from user intent to system action.

10. Summary

The Command Palette merges navigation, actions, search, context switching and shortcuts into one surface. Power users treat it as the primary workflow hub. New users use it to learn the system. In VDS, it defines how a user controls the entire product—efficiently and without friction.

Layout & structure

The command palette is a viewport overlay (.command) containing a panel (.command__panel) with a header, scrollable body and footer. Alignment and sizing are token-driven and responsive.

  • Overlay: .command covers the viewport, provides the backdrop and controls open/closed state via .command--open. CSS also responds to [data-command-open="true"] and [aria-hidden="false"] for compatibility, but the canonical hook in VDS is the class.
  • Panel: .command__panel is the dialog surface, constrained by --command-panel-max-width and --command-panel-max-height.
  • Header: Contains the search input, optional leading icon and shortcut hint block.
  • Body: Holds multiple .command__section elements, each containing a label and .command__list.
  • Footer: Helper text and/or keyboard usage hints.
HTML 
<div class="command command--center">
    <div class="command__panel" role="dialog" aria-modal="true">
        <header class="command__header">...input + shortcut...</header>

        <div class="command__body">
            <div class="command__section">
                <div class="command__section-label">Section label</div>
                <ul class="command__list" role="listbox">
                    <li class="command__item" role="option">...</li>
                </ul>
            </div>
        </div>

        <footer class="command__footer">... helper text ...</footer>
    </div>
</div>

Use .command--center to vertically center the panel, or omit it to align to the top. On small screens the panel stretches closer to full width and height while preserving rounded corners.

Accessibility & ARIA hooks

The command palette is a modal dialog containing a keyboard-driven listbox. Styling preserves native semantics and focuses on predictable roles, labelling and focus management.

  • Modal role: Use role="dialog" and aria-modal="true" on .command__panel. Provide aria-label or aria-labelledby for the panel.
  • Single focus entry: When opening, move focus to the search input. The command palette should trap focus until closed.
  • List semantics: .command__list uses role="listbox", and .command__item uses role="option". Maintain aria-selected on the active item.
  • Keyboard navigation: Support ↑/↓ to move between items, Enter to run the selected command, Esc to close, and optionally Tab to move between sections.
  • Backdrop semantics: The backdrop is decorative; do not assign interactive roles. The dialog itself handles interaction and focus.
  • Announcements: When no results are available, replace list content with .command__empty and ensure screen readers perceive that the list contents changed (e.g., via aria-live region in your JS integration if necessary).
  • Reduced motion: command.css honours prefers-reduced-motion: reduce by disabling transitions for users who opt out of motion.
HTML 
<div class="command command--center" data-command="basic">
    <div class="command__panel"
         role="dialog"
         aria-modal="true"
         aria-label="Command menu">

        <header class="command__header">
            <input
                class="command__input"
                type="search"
                aria-label="Search commands or go to…" />
        </header>

        <div class="command__body">
            <div class="command__section">
                <div class="command__section-label" id="section-nav">
                    Navigation
                </div>

                <ul class="command__list"
                    role="listbox"
                    aria-labelledby="section-nav">

                    <li class="command__item command__item--active"
                        role="option"
                        aria-selected="true">...</li>

                </ul>
            </div>
        </div>

    </div>
</div>

The JavaScript integration must manage focus trapping and keyboard event handling; the CSS layer is built to respect these semantics but does not implement them.

Integration rules & recommendations

Combined guidance from v1’s ARIA notes and v2’s usage rules. Treat the command palette as a modal dialog with listbox semantics, fully navigable via keyboard, and wired into your app’s routing and action systems.

ARIA & keyboard

  • Dialog semantics Root panel uses role="dialog" with aria-modal="true" and a clear label via aria-label or aria-labelledby.
  • Listbox semantics Result lists use role="listbox" and items use role="option" with aria-selected.
  • Keyboard navigation ↑/↓ move the active option, Enter runs it, Esc closes the palette. Maintain the active state via an internal index or aria-activedescendant.
  • Focus management When opening, move focus into the search input. When closing, return focus to the element that triggered the palette.

Integration & usage

  • Single controller Centralise all keyboard shortcuts and command-opening logic in one controller instead of distributing listeners across components.
  • Real commands, not placeholders Each item must map to an actual navigation or action handler. The palette is an execution surface, not a static menu.
  • Context-aware variants Use variant keys such as command-basic, command-nav and other scope-specific identifiers for global vs. contextual palettes.
  • Respect reduced motion If you add animations on top of command.css, honour prefers-reduced-motion and avoid intensive transitions for users who prefer minimal motion.

The command palette is a primary interaction surface in modern SaaS products. Treat it as a first-class part of your shortcut map and information architecture, not a decorative overlay.

Accessibility & usage rules

Contract for safe, semantic use of the command palette: modal dialog semantics, keyboard-driven listbox navigation, predictable focus management, and explicit state handling through command--open and ARIA attributes. Commands must be real, actionable items surfaced through the unified global interface, not cosmetic placeholders.

Do

  • Use real dialog semantics on .command__panel: role="dialog" + aria-modal="true". Provide a label via aria-label or aria-labelledby.
  • Move focus into the .command__input the moment the palette opens. This defines the palette as a single-focus-entry modal.
  • Use role="listbox" on .command__list and role="option" on .command__item. Maintain aria-selected on the active item.
  • Reflect keyboard highlight using .command__item--active and keep it in sync with aria-selected="true".
  • Let the JS controller toggle open/closed state through .command--open. The CSS also supports [data-command-open="true"] and [aria-hidden="false"].
  • Use .command__empty on no-results and hide list sections from screen readers.
  • Ensure full keyboard support: ↑/↓ navigate, Enter runs, Esc closes, Tab cycles sections (optional).
  • Preserve :focus-visible outlines. They are part of the accessibility contract.
  • Use .command--compact for dense UIs. Spacing tightens; hit targets remain accessible.
  • Keep shortcuts and metadata inside .command__item-meta.
  • Place the palette above the UI with --z-modal.

Don’t

  • Don’t fake interactivity with plain <div>. Use role="option" consistently.
  • Don’t remove or suppress :focus-visible. The palette relies on it for navigation state.
  • Don’t embed logic inside CSS. Behaviour belongs to your JS controller.
  • Don’t allow more than one “active” item. Maintain a single active index.
  • Don’t show .command__empty alongside list sections.
  • Don’t close the modal without returning focus to the element that opened it.
  • Don’t treat the backdrop as interactive. .command::before is decorative.
  • Don’t rely on colour alone for meaning. Use labels + descriptions.
  • Don’t nest another modal inside the command palette. 
1. .command__panel uses role="dialog" and aria-modal="true".
2. Focus moves to .command__input when opening.
3. Lists use role="listbox"; items use role="option" + aria-selected.
4. Visual state = .command__item--active kept in sync with aria-selected.
5. Toggle open state via .command--open only.
6. .command__empty replaces listbox in no-results mode.
7. Keyboard: ↑↓ navigate · Enter run · Esc close · Tab cycle.
8. Restore focus to trigger when closing.
9. Never remove focus-visible outlines.
