Advanced Form Controls

Extended input patterns for complex SaaS interfaces: multi-select chips, tag inputs, choice buttons, choice chips, segmented controls, toggles, numeric steppers, native date/time normalisation, and a flexible calendar scaffold for custom JS-driven pickers.

Source: forms-advanced.css · Depends on forms.css, base.css, typography.css, icons.css.

Choice buttons

Button-like radio and checkbox groups. Works with native inputs and keyboard focus.

Installation & dependencies

Load forms-advanced.css after the core form system. All advanced controls inherit tokens, density and semantics from forms.css.

  • Load forms.css first.
  • Then load forms-advanced.css.
  • Native inputs remain semantic; JavaScript is responsible for behaviour only.
  • All components respond to .form-field--error, ARIA validation, and density layers (.form--a / default / .form--c).
HTML 
<link rel="stylesheet" href="../css/forms.css">
<link rel="stylesheet" href="../css/forms-advanced.css">

Accessibility & usage rules

Contract for safe, semantic composition of advanced form controls: multi-select chips, tag inputs, steppers, native date/time inputs, and the calendar scaffold. Maintain native HTML semantics, expose state through ARIA and token-driven classes, and keep interactive behaviour delegated to real elements and JavaScript.

Do

  • Use native inputs (<input>, <select multiple>, <button>) as the semantic foundation for all advanced controls.
  • Wrap each control in a .form-field to inherit density, validation state, labels, help text, and error descriptions.
  • Connect chips and removable items with aria-label or aria-labelledby when their text alone does not express purpose (e.g., “Remove tag: X”).
  • Expose loading, error, and disabled states through .form-field--error, .multi-select--loading, [disabled] or .stepper--disabled.
  • Keep JavaScript responsibilities minimal: toggle dropdowns, update hidden inputs, manage chip lists, apply --open, --selected and similar state classes.
  • Use semantic tokens for meaning and status: color-border-subtle, color-danger, focus-ring-color.
  • Preserve calendar semantics by modelling days as interactive elements (<button> recommended) when binding behaviour through JavaScript.
  • Let density layers (.form--a, default, .form--c) adjust spacing and sizing automatically.

Don’t

  • Don’t replace real inputs with <div> for form behaviour — always wrap native controls and enhance them.
  • Don’t hard-code colours, radii or spacing for chips, steppers, dropdowns or calendars — rely strictly on VDS tokens.
  • Don’t suppress :focus-visible styles for any interactive part of multi-selects, chips, tag inputs, or calendar days.
  • Don’t encode validation state with custom colours or ad-hoc classes — use .form-field--error or the relevant --error variants.
  • Don’t embed layout logic (grid, spacing, stack) inside the advanced control itself — keep them content-neutral and let utilities handle structure.
  • Don’t mix multiple input models inside a single control (e.g., chips that behave like radios, or tag inputs that behave like lists).
  • Don’t make calendar days clickable by styling a plain <div> without roles or keyboard accessibility — use real interactive elements when behaviour exists. 
1. Use real inputs and buttons; enhance them with wrappers, never replace them.
2. Wrap everything in .form-field to inherit labels, help, and error state.
3. Expose behaviour state with ARIA and tokens (--open, --selected, --error).
4. Preserve focus-visible; never remove keyboard cues.
5. Keep layout neutral; rely on utilities for spacing and grid.
6. Surface validation exclusively via .form-field--error and semantic tokens.
7. Use JS only for behaviour (selecting, toggling, updating values), not styling.
