Accordion

Native <details>-based disclosure blocks for compact, accessible vertical sections in settings pages, documentation, dashboards, and dense configuration flows.

Source: accordion.css. Minimum runtime dependencies: base.css, one active theme in /themes/*.css, and icons.css only if you use VDS icon utilities for the chevron. This documentation page also uses layout.css, typography.css, sections.css, content-blocks.css, utilities.css, and doc-block.css for presentation.

VDS keeps the implementation intentionally native: each item stays a real <details>, the [open] attribute remains the state source of truth, richer headers use optional text slots instead of custom widgets, and local overrides can scope to .accordion, .vds-accordion, or [data-vds-accordion] through --accordion-* variables.

If you want "only one panel open" behavior, modern browsers can do it natively with a shared name attribute on sibling <details> elements. If your support matrix excludes that feature, mirror the same behavior by syncing the open attribute with a tiny script instead of replacing the component.

Installation & dependencies

accordion.css is intentionally thin. Load the component after the base layer and a single active theme. Only load icons.css if you want the VDS icon helpers for the chevron.

Runtime

  • Load base.css first.
  • Choose exactly one active theme from /themes/*.css.
  • Load icons.css only if you use icon utility classes.
  • Then load accordion.css.

Anatomy & markup contract

The structure stays intentionally small. The only required pieces are the accordion wrapper, a <details> item, a <summary> header, a title, an icon, and a body. Richer header text is optional.

Structural elements

.accordion
Vertical stack and default token scope for the component.
.accordion-item (<details>)
Single disclosure block. The native [open] attribute controls state.
.accordion-header (<summary>)
Visible toggle row. Keep it native.
.accordion-header-text
Optional multi-line wrapper for eyebrow, title, description, and meta text.
.accordion-eyebrow
Optional overline for category or grouping context.
.accordion-title
Required visible label for the item.
.accordion-title--truncate
Optional single-line truncation helper for constrained layouts.
.accordion-description / .accordion-meta
Optional secondary header lines.
.accordion-icon
Chevron indicator. Usually paired with VDS icon utilities.
.accordion-body
Expanded content region below the header.
name on <details>
Optional native exclusive-group hook for modern browsers.

<div class="accordion">
    <details class="accordion-item">
        <summary class="accordion-header">
            <span class="accordion-header-text">
                <span class="accordion-eyebrow">Optional category</span>

                <span class="accordion-meta">Optional small status line</span>
            </span>
            <span class="accordion-icon icon icon--mask icon--chevron-right"></span>
        </summary>
        <div class="accordion-body">
            <p>Disclosure content lives here.</p>

Exclusive groups with native name

When your support matrix allows it, sibling <details> items can share the same name value to create an exclusive accordion where opening one item closes the others.

How do I rotate API keys?

Create a replacement key first, update consumers, then revoke the old key.

Where can I review active sessions?

Active sessions live in the Security section of the account settings page.

Can invoice emails go to a finance alias?

Yes. Use the Billing section to set a dedicated destination address.

Shared name is the cleanest native exclusive-group pattern. If older browsers are part of your support matrix, keep the markup and synchronize open with a minimal script instead.

Usage patterns & best practices

Use accordions for progressive disclosure in dense layouts, not as a replacement for tabs or navigation. Keep headers short, content grouped, and behavior native.

Do

  • Use accordions for secondary content: advanced settings, explanations, optional configuration, or FAQs.
  • Keep header labels short and descriptive; treat them like section labels, not paragraphs.
  • Use .accordion-header-text when you need a small amount of supporting context in the header.
  • Use semantic variants only when a section carries a real meaning such as billing, warning, or destructive actions.
  • Use local --accordion-* variables for spacing or visual adjustments before forking the CSS.
  • Use shared name values when you want a native exclusive accordion on modern browsers.

Do not

  • Do not use accordions as primary navigation; use tabs, sidebar navigation, or page sections instead.
  • Do not hide critical destructive actions only inside closed items; surface the risk clearly when the section opens.
  • Do not pack long instructions, buttons, links, or form controls into the header itself.
  • Do not rely on truncation unless the layout truly requires a single visible line.
  • Do not nest accordions inside accordions unless there is a very strong information architecture reason.
  • Do not replace the native structure with arbitrary <div>s just to chase custom keyboard behavior.

Accessibility & usage rules

Accessibility comes from keeping the structure native. Preserve the <details>/<summary> contract, keep the visible label inside the summary, and let the body hold the richer interaction.

Contract

  • Always use real <details> as the root of each accordion item. State belongs on [open].
  • Keep <summary> as the visible toggle. The title, eyebrow, description, meta, and icon are fine inside it.
  • Keep links, buttons, form fields, and other persistent interactive controls out of <summary>; place them in the body or an adjacent toolbar instead.
  • Do not remove or override the component's :focus-visible treatment.
  • If the page needs formal document-outline headings, place them around the accordion sections rather than relying on heading elements inside <summary>.
  • Avoid fixed heights or clipping on the body; disclosure content needs room to grow.

Keyboard & AT

  • Native disclosure elements expose expanded and collapsed state to assistive technology.
  • Keyboard users toggle items with Enter or Space.
  • Tab and Shift + Tab move through the summary and any focusable content inside the expanded body.
  • Arrow-key, Home, and End roving behavior is not provided here because VDS intentionally keeps the pattern native.
  • Shared name values can provide native exclusive-group behavior, but older-browser support should be handled explicitly if required.
  • Keep the visible title short and specific because it acts as the primary spoken label for the disclosure.
