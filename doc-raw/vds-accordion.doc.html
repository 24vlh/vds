Accordion

Native <details>-based disclosure blocks for compact, accessible vertical sections in settings pages, documentation, and dense UIs.

Source: accordion.css · Depends on base.css, layout.css, typography.css, icons.css, and theme tokens in /themes/*.css.

This page documents the full accordion surface: native <details> anatomy, header and body roles, chevron icons, open/closed state, semantic variants, and accessibility rules. Use it for progressive disclosure where tabs would be too heavy and inline content would be too long.

Installation & dependencies

accordion.css is a thin component layer for native disclosure blocks. Load it after core layout, typography, icons, and theme tokens. Documentation-only styles remain optional.

  • Load base.css and layout.css first.
  • Load typography.css and icons.css.
  • Load an active theme: themes/slate.css, etc.
  • Then load accordion.css.
  • Optionally load doc-block.css.

Anatomy & markup contract

Accordions in VDS are built on native <details> elements. Each item is a disclosure block with a clickable header and a body region.

Structural elements

.accordion
Vertical stack of accordion items.
.accordion-item (<details>)
Single disclosure block.
.accordion-header (<summary>)
Clickable header row.
.accordion-title
Header text span.
.accordion-icon
Chevron indicator.
.accordion-body
Content region.
[open]
Native <details> state attribute.
HTML 
<div class="accordion">
    <details class="accordion-item">
        <summary class="accordion-header">
            <span class="accordion-title">Section title</span>
            <span class="accordion-icon icon icon--mask icon--chevron-right"></span>
        </summary>
        <div class="accordion-body">
            <p>Disclosure content lives here.</p>
        </div>
    </details>
</div>

States & variants

Accordions support a neutral base style and semantic variants that reuse existing feedback tokens. Open state adds an accent border-left; chevron rotation indicates expansion. There are no height animations.

Base vs semantic variants

Neutral item (open)

Uses --color-accent for the left border when open.

Info variant

Uses --color-info for the open border-left.

Success variant

Uses --color-success for the open border-left.

Density layers

Accordions adapt to VDS density layers by adjusting header and body padding. Content stays identical; only vertical rhythm changes to match the surrounding layout.

Usage patterns & best practices

Use accordions for progressive disclosure in dense layouts, not as a replacement for tabs or navigation. Keep headers clear, body content short, and avoid nesting too deeply inside other collapsible components.

Do

  • Use accordions for secondary content: advanced settings, explanations, and optional configuration blocks.
  • Keep header labels short and descriptive; treat them like section titles, not full sentences.
  • Group related content into one accordion item instead of many tiny ones.
  • Use semantic variants to draw attention to important sections (e.g. danger zone, billing, legal).
  • Respect native keyboard behaviour; headers remain real <summary> elements.

Don’t

  • Don’t use accordions as primary navigation; use tabs, sidebar nav, or breadcrumbs instead.
  • Don’t place critical blocking actions (like “Delete account”) inside closed items only; surface them clearly.
  • Don’t nest accordions inside accordions; prefer a single-level structure with clear sections.
  • Don’t remove focus styles or override :focus-visible on headers or actions.
  • Don’t fake the markup with arbitrary <div>s; rely on real <details>/<summary>.

Accessibility & usage rules

Accordions rely on native <details>/<summary> semantics, keyboard focus, and token-driven focus styles. Keep the structure intact to preserve assistive technology behaviour.

Contract

  • Always use real <details> as the root of each accordion item. The [open] attribute controls state.
  • Keep <summary> as the header; do not replace it with non-semantic elements.
  • Do not remove or override :focus-visible styles on .accordion-header.
  • Avoid JS that breaks native behaviour; if you need “only one open at a time”, toggle the open attribute.
  • Ensure the body remains readable; avoid fixed heights or clipping.

Keyboard & AT

  • Native disclosure elements expose expanded/collapsed state to assistive tech.
  • Keyboard users toggle with Enter or Space.
  • The visible header must always remain the toggle element.
  • Titles should be short; they act as labels for screen readers.
