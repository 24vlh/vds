Authoring Layer

authoring.css provides the authoring surface for VDS: admonitions, MDX-style callouts, heading anchors, footnotes, inline markers and prose helpers that sit cleanly on top of your typography.

Source: authoring.css · Safe on top of primitives.css, themes/slate.css, base.css, layout.css and typography.css.

This page documents the full authoring surface: base admonitions, Markdown callouts, heading anchors, footnotes, inline markers, prose helpers and doc-block integration. Everything is designed to work with semantic HTML and the existing VDS documentation stack.

Installation & dependencies

authoring.css is a thin layer that styles semantic documentation markup. It assumes tokens, theme, base, layout and typography are already loaded.

  • Load neutral tokens: primitives.css.
  • Load one active theme, e.g. themes/slate.css.
  • Load core structural layers: base.css, layout.css, typography.css.
  • Optionally load: utilities.css, content-blocks.css, doc-block.css.
  • Then load authoring.css to enable admonitions, callouts, anchors, footnotes and prose helpers.

Authoring scope & foundation

A single class or data attribute sets sensible defaults for admonitions, footnotes and inline markers. Use it on any container that hosts documentation content.

Scope helpers

Wrap long-form documentation areas with .authoring, .doc-authoring, [data-vds-authoring] or .vds-authoring to initialise authoring variables.

Authoring context
This box lives inside an .authoring container, so all authoring tokens are active.

You can scope the authoring layer to a single article or the whole page. 

<article class="typography" data-vds-authoring>
  <h1>Guide title</h1>
  <p>Introductory copy…</p>

  <div class="admonition admonition--note">
    <div class="admonition__icon"><span class="icon icon--mask icon--info"></span></div>
    <div class="admonition__content">
      <div class="admonition__title">Note</div>
      <div class="admonition__body">
        This guide assumes Node 20+ is installed.
      </div>
    </div>
  </div>
</article>

Heading anchors

Small, hover-revealed anchors for deep-linking into sections. Works with plain headings or headings inside a .typography container. On touch devices anchors stay visible; use .heading-anchor--always to keep them persistent.

Hover to reveal

Setup #

Hover or focus the heading to reveal the # anchor. The anchor uses theme colours and respects focus states.

Attach id to the heading and point the anchor’s href to the same value for deep-linking. 

<article class="typography authoring">
  <h2 id="setup">
    Setup
    <a
      href="#setup"
      class="heading-anchor"
      aria-label="Link to this section"
    >#</a>
  </h2>

  <p>Use anchors for shareable section links in long guides.</p>
</article>

Authoring + doc-block integration

The authoring layer is tuned to sit inside .doc-block bodies: margins are normalised so multiple callouts do not break the documentation rhythm.

Composed inside a doc-block

You can safely mix admonitions, Markdown callouts, inline markers and footnotes inside the same documentation block.

Authoring-aware spacing
When used inside .doc-block__body, authoring.css slightly adjusts vertical margins to match the documentation grid.

Tip. Use numbered code blocks from doc-block.css together with inline markers to explain each step.

Numbered lines are ideal for step-by-step explanations see example below.

Accessibility & usage rules

Contract for accessible, predictable authoring surfaces. Maintain semantic HTML, respect typography, and let tokens drive spacing, colour and hierarchy.

Do

  • Wrap documentation areas in .authoring, .doc-authoring, .vds-authoring, or [data-vds-authoring] to activate tokens.
  • Use semantic structure for admonitions: .admonition, .admonition__title, .admonition__body.
  • Use MDX-style callouts (.md-note, .md-warning, etc.) only when the content comes from Markdown. Prefer .md-callout + data-variant for pipelines that need a single class.
  • Attach anchors directly to headings (h2–h6) with matching id and href.
  • Use .fn-ref and .fn-back for proper footnote semantics.
  • Use inline markers (.inline-marker--info, .inline-marker--success, .inline-marker--warning, .inline-marker--danger, .inline-marker--accent, .inline-marker--neutral) for short, high-signal inline labels only.
  • Use data-variant or semantic modifiers for colours; keep the palette token-driven.
  • Use .prose-tight / .prose-relaxed to tune reading rhythm without overriding typography.
  • Compose authoring primitives inside .doc-block__body for consistent margins.

Don’t

  • Don’t nest admonitions or MDX callouts inside paragraphs.
  • Don’t mix .admonition and Markdown callouts (.md-note, .md-info, .md-tip, .md-success, .md-warning, .md-danger, .md-important) in the same block; choose one model per context.
  • Don’t place heading anchors outside the heading or after sibling elements.
  • Don’t use footnotes for clarifications better served by admonitions.
  • Don’t overuse inline markers; they are for emphasis, not decoration.
  • Don’t apply authoring styles globally to application UI surfaces.
  • Don’t override semantic colours of admonitions or markers; rely on theme tokens.
1. Wrap documentation content in .authoring, .doc-authoring, .vds-authoring, or [data-vds-authoring].
2. Use .admonition or Markdown callouts (.md-note, .md-info, .md-tip, .md-success, .md-warning, .md-danger, .md-important) consistently (never mixed); prefer .md-callout + data-variant for pipelines.
3. Place heading anchors directly in h2–h6 with matching id/href.
4. Keep footnotes semantic: .fn-ref in copy, .footnotes list below.
5. Use inline-marker--info|success|warning|danger|accent|neutral (or data-variant) only for short, high-signal labels.
6. Preserve spacing via prose-tight / prose-relaxed; do not override.
7. Never apply global authoring styles to UI components or layouts.
