Feedback System

The VDS Feedback Layer provides alerts, banners, feedback-toasts, status indicators, info blocks, progress visualizations and result components. All are fully token-driven, theme-aware, and consistent with VDS semantics.

Source: feedback.css · Sits above primitives.css, themes/slate.css, base.css, layout.css and utilities.css.

This page documents all primitives exported by feedback.css. Components follow a consistent structure and use semantic variants: info, success, warning, danger, error, neutral and accent.

Overview & Dependencies

feedback.css relies fully on VDS token layers: spacing, typography, radius, shadows, border-width, semantic colors and transitions. It introduces no globals and cleanly composes with base, layout, utilities and doc-block layers.

  • All colors come from semantic tokens in your theme.
  • Spacing uses --space-* primitives.
  • Transition speeds use var(--transition-normal).
  • Border-radius uses var(--radius-sm/md).
  • All dismissible components use accessibility-safe focus-visible states.
  • Density modes apply uniformly across all primitives.

Feedback Primitives & Design Rules

Each feedback primitive follows a predictable API: structural wrapper, optional icon, content block, optional close button, semantic variants and density control.

  • Alerts – inline, dismissible, semantic variants.
  • Banners – full-width horizontal notifications.
  • feedback-toasts – ephemeral notifications with animation + stack.
  • Guidance Blocks – do/don't/how panels for instruction-heavy flows.
  • Info Blocks – informational panels with left-border accent.
  • Status Indicators – inline pills, outline mode, and dot mode.
  • Progress Bars – striped / animated options.
  • Result Blocks – centered success/error panels with actions.
  • Density System – compact mode across all primitives.

Guidance Blocks

Lightweight instructional panels for do/don't/how guidance that sit next to forms without adding visual weight.

Do
  • Lead with the hook in one sentence.
  • Keep beats to 3–6 clear steps.
Don't
  • Avoid long preambles.
  • Don't merge unrelated ideas.
How
Write beats as short verbs: "Name, explain, prove, recap."

Accessibility & usage rules

Contract for accessible, semantic, token-driven feedback components. Ensure alerts, banners, feedback-toasts, info blocks, indicators, progress bars and results remain navigable, readable and meaningful to assistive technology.

Do

  • Use semantic variants (data-variant) to convey real meaning: info, success, warning, danger, error, neutral, accent.
  • Always provide accessible text labels for dismiss buttons: aria-label="Close …".
  • Use alerts for inline messages, banners for top-of-context notices, and feedback-toasts for ephemeral notifications.
  • Use status for inline semantic cues and progress for quantitative state, never interchange these roles.
  • Ensure feedback-toasts remain keyboard-accessible: dismiss buttons must be real <button> elements with focus-visible.
  • Use data-density="compact" for reduced-spacing interfaces; apply at container scope.
  • Keep animations token-driven and respect user motion preferences (the layer already supports reduced-motion).
  • Place result blocks in contexts where a user expects a stop or confirmation moment (e.g., success, error, completion flows).

Don’t

  • Don’t repurpose danger or warning variants for neutral or low-risk events.
  • Don’t hide validation or warning messages behind hovers, popovers or tooltips—feedback must be directly visible.
  • Don’t place banners mid-content; they must anchor to the top of the container or page.
  • Don’t overload feedback-toasts with long content, links or multi-action blocks.
  • Don’t use outline mode (alert--outline, status--outline) unless visually appropriate for the surrounding UI density.
  • Don’t animate progress bars or feedback-toasts aggressively; keep motion subtle and token-consistent.
  • Don’t mix multiple conflicting variants in the same structural element (e.g., data-variant="success warning").
  • Don’t override semantic colors with custom inlined colors—theme tokens must remain the single source of truth. 
1. Use data-variant for real semantic meaning (info/success/warning/danger/etc.).
2. Provide aria-labels for all dismiss buttons in alerts, banners and feedback-toasts.
3. Use banners only at the top of contexts; use feedback-toasts only for ephemeral events.
4. Keep alerts inline and succinct; use status for inline semantic cues.
5. Preserve token-driven motion, spacing, borders and colors.
6. Apply data-density="compact" on containers, not per-element.
7. Do not override focus-visible or interactive hit-areas.
8. Keep feedback-toasts short-lived, accessible, and dismissible via keyboard.
