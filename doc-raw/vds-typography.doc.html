Foundations

VDS Typography

Token-driven type system for headings, body, metadata and long-form prose. Components do not invent type; they compose this layer.

Source: typography.css · Depends on base.css, primitives.css, and one active theme in /themes/*.css.

Layer stack

Typography operates on top of primitives, base and theme tokens. It does not define layout or utilities. Load in this order.

  • primitives.css — core scales (type, spacing, radii).
  • base.css — resets, body, generic text helpers.
  • typography.css — headings, paragraphs, prose layout.
  • themes/slate.css (or other) — colors and contrast.
  • sections.css / content-blocks.css — used here for documentation layout only.

Heading hierarchy

Semantically correct headings (h1–h6) with tokenised sizes and consistent spacing. First heading inside .typography has its top margin collapsed.

H1 · Page / document title

Used once per page / document. Typically rendered by layout, not inline content.

H2 · Section title

Major sections of a page. Think “chapters” in documentation.

H3 · Subsection title

Use for important subtopics inside an H2 section.

H4 · Minor heading

Sub-headings within an H3 area, often used with smaller blocks.

H5 · Label heading

Useful for inline labels, sidebars, or metadata sections.

H6 · Small heading

Use sparingly. Often replaced by .text-xs labels instead.

<div class="typography">
    <h1>H1 · Page / document title</h1>
    <p>Used once per page / document.</p>

    <h2>H2 · Section title</h2>
    <p>Major sections of a page.</p>

    <h3>H3 · Subsection title</h3>
    <p>Subtopics inside an H2 section.</p>

    <h4>H4 · Minor heading</h4>
    <h5>H5 · Label heading</h5>
    <h6>H6 · Small heading</h6>
</div>

Body text & rhythm

Paragraphs use base font size and relaxed line height. Vertical rhythm is controlled via tokens rather than ad-hoc margins.

This is a standard paragraph inside .typography. It uses the base font size, a balanced line height and tokenised vertical spacing.

Subsequent paragraphs maintain consistent rhythm. Do not add manual margins between paragraphs; rely on the system spacing instead.

Use .text-sm or .text-xs for metadata, labels or supporting descriptions. 

<div class="typography">
    <p>
        This is a standard paragraph inside <code>.typography</code>.
        It uses the base font size and tokenised vertical rhythm.
    </p>

    <p>
        Subsequent paragraphs use the same spacing. Avoid manual margins.
    </p>

    <p class="text-sm text-muted">
        Use smaller sizes for metadata or helper text.
    </p>
</div>

Inline semantics

Semantic tags are styled minimally: <strong>, <em>, inline <code> and helper badges via .badge*.

Strong text is used for emphasis, emphasis text for nuance. Combine them sparingly.

Inline code like npm run build or POST /api/v1/orders uses the mono font and a subtle tokenised background.

You can embed beta, info, success, warning, danger badges directly into prose. 

<div class="typography">
    <p>
        <strong>Strong</strong> and <em>emphasis</em> text.
    </p>

    <p>
        Inline code like <code>npm run build</code> or

Accessibility & usage rules

Typography is semantic-first: headings, paragraphs, lists, tables and code blocks must use correct HTML elements. Appearance is token-driven; readability, rhythm and structure come from real semantics, not classes. Contrast and legibility are theme-controlled.

Do

  • Use semantic HTML: <h1–h6>, <p>, <ul/ol/li>, <blockquote>, <figure>, <figcaption>, <pre><code>.
  • Maintain heading hierarchy — one <h1> per document, <h2> for major sections, <h3> for subsections, etc. Typography tokens handle visual size; headings handle structure.
  • Wrap long-form content in .typography to access spacing, list, prose-table, code block and footnote rules.
  • Use <caption> for prose tables that require explanation or context.
  • Keep inline code meaningful — short technical identifiers only. Rendering uses mono font + subtle background/border tokens.
  • Use proper list semantics; nested lists inherit token-driven spacing automatically.
  • Treat <blockquote> as quoted content with attribution; use .pullquote only for editorial emphasis.
  • Keep figure captions tied to images and diagrams via <figcaption>.
  • Maintain token-driven readability: no manual margins, font-sizes, or ad-hoc spacing.
  • Use footnotes with .fn-ref and .footnotes; links must preserve bidirectional navigation.
  • Ensure contrast tokens from active theme provide accessible text/background levels.

Don’t

  • Don’t use headings (<h3>, <h4>, …) to achieve visual size — respect document outline.
  • Don’t place component tables (<table class="table">) inside .typography expecting prose styling — they are owned by tables.css.
  • Don’t style text with arbitrary inline CSS (font-size, color, margin). Use tokens and the existing classes.
  • Don’t wrap code blocks in arbitrary containers that break scrolling or token padding.
  • Don’t use blockquotes as layout tools — they imply quotation semantics.
  • Don’t override inline code styles inside prose tables manually. Component tables already disable code formatting intentionally.
  • Don’t fake footnotes by manually shrinking text; use the provided .footnotes, .fn-ref, and .fn-backref patterns.
  • Don’t mix layout concerns (columns, grids, flex) inside .typography; keep layout in layout/sections layers.
  • Don’t rely on colour alone for meaning — emphasize with text, badges, or structural elements. 
1. Use semantic HTML for all text elements.
2. Maintain a strict heading hierarchy; visual scale is token-driven.
3. Wrap long-form content in .typography for rhythm & prose rules.
4. Prose tables only: component tables belong to tables.css.
5. Inline code = short identifiers; blocks = pre > code.
6. Use figure/figcaption for media requiring explanation.
7. Footnotes require fn-ref, footnotes, fn-backref.
8. Never override typographic tokens manually.
9. Typography layer contains no layout logic; keep structure semantic.
