Header & Footer

Site-wide chrome: executive header, sticky elevation, integrated navigation, and structured footers for legal, multi-column and compact layouts — all theme-driven.

Source: header-footer.css · Depends on base.css, typography.css, navigation.css, theme tokens in /themes/*.css.

Installation & dependencies

The header/footer frame sits on top of base layout, typography and navigation primitives. Load CSS in this order to avoid override conflicts.

  • base.css — reset, tokens, z-index hooks.
  • layout.css — page layout container helpers.
  • typography.css — type scale.
  • navigation.css — nav/tabs/pagination primitives.
  • header-footer.css — this subsystem.
  • One active theme: themes/slate.css, themes/navy.css, themes/carbon.css.

Structural model

Header and footer are thin chrome layers around the content column. Both share the same token-driven layout principles and theme-bound colours.

Header primitives

  • .header — global header container.
  • .header--sticky — sticks to top, with background + border.
  • .header--elevated — additional shadow/elevation.
  • .header--compact, .header--tall — density control.
  • .header--borderless, .header--transparent — chrome variants.
  • .header__inner — flex row for left/right regions.
  • .header__left, .header__right, .header__actions — layout helpers.
  • .header__toggle — mobile menu toggle button.
  • .header--nav-open .nav — mobile nav expansion.
  • .header__panel + .header--nav-panel — dedicated mobile panel.

Footer primitives

  • .footer — global footer container.
  • .footer__inner — content layout row/stack.
  • .footer__columns — responsive column grid.
  • .footer__col — column blocks.
  • .footer__col-title — column label text.
  • .footer__link — footer navigation links.
  • .footer__meta — legal / meta text block.
  • .footer__bottom, .footer__actions — meta + actions row.
  • .footer--compact, .footer--borderless — density variants.

Skip-link & access helpers

The header/footer layer includes a global skip-link pattern to jump straight to main content for keyboard and screen-reader users.

Skip-links are visually hidden until focused, then presented clearly near the header.

Header CSS defines .skip-link and .skip-link:focus-visible styles. 

<body>
    <a class="skip-link" href="#main">
        Skip to main content
    </a>

    <header class="header">...</header>
    <main id="main">...</main>
</body>

Accessibility & usage rules

Header and footer surfaces expose explicit landmarks, keyboard-first navigation, theme-driven contrast guarantees, and mobile-safe toggle mechanics. These rules ensure predictable behaviour across all layouts, themes and viewport ranges.

Do

  • Use semantic landmarks: <header role="banner">, <main role="main">, <footer role="contentinfo">.
  • Keep the skip-link active and focusable — the .skip-link pattern is part of the system and must resolve to #main or an equivalent main landmark.
  • Provide aria-label and aria-controls on .header__toggle for mobile nav expansion; update aria-expanded when controlling via JS.
  • Use .header--nav-panel with .header__panel when you render a dedicated mobile nav.
  • Use :focus-visible outlines from theme tokens on header and footer links, navigation items, and toggles.
  • Ensure navigation is operable entirely via keyboard: Tab → navigate items, Enter/Space → activate, Esc → close mobile menu (if using JS).
  • Keep contrast at theme-safe levels via token-bound colours: --color-header-bg, --color-footer-text, --color-accent, --focus-ring-color.
  • Maintain predictable reading order — header → skip-link → nav → content → footer. Avoid DOM reordering for visual layouts.
  • Expose active navigation state via: .nav__item--active AND (optionally) aria-current="page" for screen readers.
  • Collapse footer columns responsively: the system already defines a mobile breakpoint at 768px; do not override with fixed px.
  • Wrap pages in .shell to keep the footer anchored on short pages.

Don’t

  • Don’t remove :focus-visible outlines from links or toggles — accessibility fails and violates system rules.
  • Don’t use custom colours, shadows or borders inside .header or .footer. These surfaces must remain 100% theme-controlled.
  • Don’t hide navigation items at mobile breakpoints via display:none; rely on the .header--nav-open mechanism.
  • Don’t show both desktop and mobile navs at once; use .header--nav-panel for separate mobile panels.
  • Don’t place oversized interactive widgets in the header — maintain header height via --header-height and alignment rules.
  • Don’t duplicate the same navigation tree across desktop/mobile without ensuring consistent labelling and ordering.
  • Don’t inject role overrides (e.g. role="navigation") on individual nav links; apply ARIA at container level only.
  • Don’t compress footer link hit-areas below theme token spacing; maintain vertical rhythm and touch-safe padding. 
1. Provide real landmarks: header (banner), main, footer (contentinfo).
2. Keep .skip-link present, focusable, and pointing to #main.
3. Header toggle: aria-label, aria-controls, aria-expanded (JS-updated).
4. Use .header--nav-panel + .header__panel for dedicated mobile nav.
5. Preserve theme-driven focus-visible and contrast tokens everywhere.
6. Use .nav__item--active + aria-current="page" for active nav state.
7. Mobile nav: toggle via .header--nav-open (CSS) or aria-expanded (JS).
8. Footer: maintain column/stack behaviour at 768px breakpoint.
9. Wrap pages in .shell to keep footer anchored on short content.
10. No custom colours, borders, or shadows; theme owns all visual tokens.
