Flows

System components for multi-step experiences, journeys, timelines, branching decisions and progress indicators. Built on flows.css (final version) :contentReference[oaicite:1]{index=1}.

Includes: Steps (horizontal/vertical), Progress bars, Minimal & Rich Timelines, Journey Maps, Process Flows, State Flows, Form Flows, Branching Flows, and compact/upcoming variants — all theme-aware and token-driven.

Installation & dependencies

The flows layer builds on primitives, theme, base, layout, utilities, content-blocks and doc-block layers. Load it after these.

  • Token foundation: primitives.css
  • Theme colours: themes/slate.css
  • Base + layout + typography
  • Optional: utilities.css
  • Docs engine: doc-block.css
  • Finally: flows.css

Accessibility & usage rules

Mandatory rules for ensuring semantic correctness, predictable keyboard behaviour, token-consistent contrast, and safe composability across all VDS environments.

1. Semantics

  • Use the appropriate semantic element: buttons must use <button>, links use <a>, lists use <ul/li>, steps/timelines must reflect ordered flow.
  • Avoid using div-only structures for interactive components.
  • Headings follow a real document hierarchy (H1 → H2 → H3). Do not skip levels.

2. Keyboard behaviour

  • All interactive elements require focus states derived from theme tokens.
  • Clickable modifiers (--clickable) must wrap a focusable element (<button>, <a>, tabindex="0").
  • Never hide focus outlines; rely on --focus-ring-* tokens for consistency.

3. Colour & contrast

  • Text/foreground must meet WCAG AA minimum contrast against their background.
  • State colours (success, warning, danger, info) must come from theme tokens: --color-success-*, --color-warning-*, etc.
  • Never hardcode colours; components must remain theme-portable.

4. Motion

  • Transitions must be token-based (--motion-fast, --motion-base).
  • Avoid continuous animation unless meaningful (progress, loading indicators).
  • Respect user preference: @media (prefers-reduced-motion: reduce).

5. Responsive rules

  • Layouts adapt using spacing tokens; do not introduce arbitrary pixel values.
  • Flow components (steps, timelines, journeys) must stack vertically under narrow breakpoints.
  • Grids use VDS layout breakpoints: --bp-sm, --bp-md, --bp-lg.

6. Composability

  • Only combine components that are designed to nest (e.g. steps inside flows, badges inside tables).
  • Do not manually override padding/margins; rely on section__stack-* utilities.
  • Content blocks must remain theme-agnostic; avoid fixed-width or fixed-height patterns.

7. State rules

  • Use state modifiers only as documented: --active, --complete, --warning, --error, --disabled, --upcoming.
  • Disabled items must be non-interactive and excluded from tab order.
  • Active states must be programmatically identifiable (e.g. aria-current="step").

8. Text rules

  • Avoid inline style overrides; only tokens or utility classes allowed.
  • Icons must include aria-hidden="true" unless conveying meaning.

9. Performance

  • No component may rely on heavy selectors (descendant chains, complex combinators).
  • Limit DOM nesting; flows and timeline items should remain shallow.
  • Transitions should be GPU-safe (opacity, transform).

10. Documentation alignment

  • Each component must declare its semantic role in its documentation page.
  • Include ARIA attributes where needed (progress, steps, navigation).
  • Examples must use real tokens, no hardcoded overrides.
