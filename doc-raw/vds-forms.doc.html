Installation & dependencies

forms.css sits above the base, layout, typography and theme layers. Load files in this order to guarantee correct tokens and resets.

  • Load base.css and layout.css first (page reset and layout primitives).

  • Load an active theme: themes/slate.css, themes/navy.css, or themes/carbon.css.

Unified Form Abstraction Layer (UFAL)

UFAL introduces a universal control interface (.form-control) and a structural wrapper (.form-control-wrapper) that unify all form components—inputs, textareas, selects, sliders, switches and future controls—under a single behavioural contract without changing their visual identity.

Why UFAL exists

Traditional form systems rely on per-component behaviour. UFAL provides an unified structural layer that:

  • Standardises width, layout and behavioural hooks across all controls.
  • Supports icons, validation rings, loading states and overlays consistently.
  • Allows all controls to participate in density scaling via .form--a and .form--c.
  • Decouples behaviour from component visuals (inputs, selects, textarea remain unchanged).

UFAL Anatomy

.form-control — the universal primitive

  • Takes up available width.
  • Defines unified behavioural hooks (loading, validation, disabled, readonly).
  • Never sets padding, borders, colours or typography.

.form-control-wrapper — structural container

  • Handles inline icons.
  • Hosts loading overlays.
  • Defines relative positioning for future add-ons.

UFAL & Density

UFAL respects existing VDS form densities: .form--a (comfortable) and .form--c (compact). Controls automatically adopt the correct minimum height via tokens.

Supported semantic states

  • .form-control--loading
  • .form-control--success
  • .form-control--warning
  • .form-control--error
  • .form-control--info
  • .form-control--disabled
  • .form-control--readonly
<div class="form-field">
    <label class="form-label">Email</label>
    <div class="form-control-wrapper">
        <input class="input form-control">
    </div>
    <p class="form-help">We never share your email.</p>
</div>

UFAL Migration Guide

Guidance for upgrading existing VDS form components to the Unified Form Abstraction Layer. This migration is non-breaking and requires minimal structural adjustments.

Why migrate?

UFAL introduces a single universal primitive (.form-control) and a structural wrapper (.form-control-wrapper) that consolidate behaviour across all form components. Adoption enables consistent loading, validation, density scaling, and icon placement.

Step 1 — Add .form-control to each control

If your forms currently use: 

<input class="input">
<select class="select">
<textarea class="textarea">

Upgrade to:

<input class="input form-control">
<select class="select form-control">
<textarea class="textarea form-control">

Step 2 — Wrap controls when necessary

Use the wrapper when you want icons, validation rings, or loading overlays. 

<div class="form-control-wrapper">
    <input class="input form-control">
</div>

Step 3 — Remove legacy wrappers

Replace non-VDS structural wrappers (e.g., .control, .field-wrapper) with .form-control-wrapper.

Step 4 — Adopt semantic states

Replace ad-hoc classes such as .is-error, .loading, .disabled-control with UFAL equivalents: 

.form-control--loading
.form-control--error
.form-control--success
.form-control--warning
.form-control--info
.form-control--disabled
.form-control--readonly

Step 5 — Integrate density properly

Ensure your form root is either .form--a or .form--c so the correct control height is propagated via UFAL tokens.

Migration is zero-breaking. UFAL only adds structural capabilities: controls keep their native appearance.

UFAL Best Practices & Rules

Universal guidance for producing predictable, accessible and scalable form experiences across VDS applications, dashboards and marketing surfaces.

Do

  • Always wrap inputs in .form-control-wrapper when icons, loading, or validation are present.
  • Place .form-label before the control wrapper.
  • Use .form-help for messages; avoid placing help text in labels or placeholders.
  • Use form-control--* state classes for visual feedback: error, success, info, warning, loading, disabled, readonly.
  • Prefer form--a or form--c densities at the form root, not on individual controls.
  • Keep spacing consistent using section__stack-* or stack gap-* utilities.
  • Compose complex controls (segmented, input-groups) using form-control so validation applies correctly.

Don’t

  • Don’t mix inline validation icons inside the control element.
  • Don’t apply validation colours manually (never use raw CSS colours).
  • Don’t use placeholders as labels — they are not accessible.
  • Don’t nest form controls inside semantic heading elements.
  • Don’t manually change control height, padding or border radius.
  • Don’t apply disabled and visual state classes together.
  • Don’t hide validation messages; screen readers rely on them.

UFAL Density System

Density across VDS forms is controlled exclusively at the form container level. UFAL exposes min-height tokens through form--a and form--c, guaranteeing consistent vertical rhythm across all controls without local overrides.

Density Levels

  • form--a — Application density (default). Uses --size-11.
  • form--c — Compact density, for tables, filters, toolbars. Uses --size-9.
  • No modifier — Neutral density (marketing surfaces / custom layouts).

Controls never set their own height — the form root governs all min-heights.

UFAL Accessibility (WCAG 2.2 AA)

UFAL enforces a strict accessibility contract covering semantics, focus states, labels, validation, assistive technology, keyboard navigation and visual contrast.

Labeling

  • Every control requires a visible .form-label.
  • Placeholder-only fields are strictly disallowed.

Help & Validation

  • Use .form-help for hints and error states.
  • Validation states must use semantic colours (form-control--error, --warning, --success, --info).

Focus & Interaction

  • All controls must preserve native focus outlines (never remove outline).
  • Ensure icons never intercept focus; they must be pointer-events: none.

Keyboard Navigation

  • Tab should reach all interactive elements, in logical document order.
  • Space/Enter should activate buttons and toggles.

Contrast Requirements

  • All text must pass WCAG 2.2 AA contrast.
  • Validation colours must remain visible in both surface and inverted modes.
  • Labels, help text and placeholders must never collide or overlap.

Accessibility & usage rules

Form controls rely on native semantics, explicit label binding, keyboard-first focus behaviour, and ARIA-driven validation. These rules ensure predictable, assistive-tech friendly forms across all VDS density layers and control types.

  • Always bind labels via for/id. Every control in forms.css depends on explicit label relationships; placeholder text must never act as a label.
  • Never remove :focus-visible styles. Focus rings are token-driven (--focus-ring-color) and must remain intact for keyboard accessibility.
  • ARIA validation is fully supported. aria-invalid="true" triggers the same styling as .input--error, .select--error, .textarea--error.
  • Screen-reader-only labels: Use .form-label--sr-only when the visual label would clash with layout (e.g., dense search bars), but still require accessible naming.
  • Never hide native controls via opacity hacks. Checkboxes, radios and switches are restyled but remain fully interactive native inputs. Do not replace with custom divs.
  • Input groups inherit the wrapper’s semantics. .input-group responds to .form-field--error, :focus-within and tokenised border logic. Do not wrap <input> elements in additional containers inside groups.
  • Do not remove native hit-targets. Checkbox, radio and switch sizes are tokenised (not shrunk). Altering width/height breaks accessibility.
  • Segmentation must represent mutually exclusive choices. .segmented is for 2–5 discrete options only; never use it for multi-select semantics.
  • Loading states must remain semantic. .input--loading, .select--loading, .textarea--loading must not be combined with aria-invalid="true". One state per control.
  • File uploads keep the native file input accessible. The hidden input is visually clipped, not removed. Do not change its visibility or positioning; assistive tech depends on the original element.
  • Density layers affect padding and font-size only. .form--a and .form--c must not be used to shrink functional hit areas below accessibility thresholds.
  • Form-level messages must precede interactive elements. .form-message provides context for screen readers; place it at the top of the form.
  • Field-level messages must follow the control. .form-field-message communicates state (error, warning, success, info) and must immediately follow the input/textarea/select to pair correctly with assistive technology reading order.
  • Do not rely on placeholder color for semantic meaning. Placeholders are stylistic only; for warnings/errors use labels + field messages.
  • Grid / row layouts must preserve DOM reading order. .form-grid and .form-row never reorder DOM nodes; screen readers follow document order, not visual positioning.

These rules are derived from the source implementation of forms.css (token-driven borders, radius, ARIA hooks, native controls, loading states and density layers). :contentReference[oaicite:0]{index=0}
