Documentation Blocks

doc-block.css is the documentation engine for VDS: a compact system of blocks, previews and code frames designed to teach CSS layers quickly and consistently.

Source: doc-block.css · Sits above primitives.css, themes/*.css, base.css, layout.css and typography.css.

This page documents the full authoring surface: block anatomy, surfaces, previews, code frames, numbered lines, diffs, semantic presets, and meta notes. Every documentation page in VDS composes these primitives.

Installation & dependencies

doc-block.css is a thin layer that styles semantic documentation markup. It assumes tokens, themes, base, layout and typography are already loaded.

  • Load neutral tokens: primitives.css.
  • Load one active theme: themes/slate.css, themes/navy.css, or themes/carbon.css.
  • Load core structural layers: base.css, layout.css, typography.css.
  • Optionally load: utilities.css, sections.css, buttons.css.
  • Then load doc-block.css to style documentation blocks and code surfaces.

Core concept & goals

Documentation blocks are teaching units: each block pairs an explanation with an optional preview and code frame, all on a single, readable surface.

Use .doc-block for any self-contained explanation: “what this layer is”, “how to wire it”, “why it exists”.

  • Consistent anatomy — title, summary, body, preview, code.
  • Theme-aware — surfaces and borders use semantic tokens.
  • Composition-first — pairs naturally with sections, grids, utilities.
  • Authoring-friendly — predictable classes, minimal nesting, easy to copy between docs. 
<article class="doc-block">
  <header class="doc-block__header">
    <h2 class="doc-block__title">Title</h2>
    <p class="doc-block__summary">Short explanation.</p>
  </header>

  <div class="doc-block__body">
  </div>
</article>

Anatomy of a documentation block

Each block has a small, fixed vocabulary of elements. This keeps every documentation page visually consistent, regardless of layer or component.

Structural elements

.doc-block__header
Title, summary and key notes.
.doc-block__body
Explanations, lists, tables, previews and code.
.doc-block__preview
Visual examples: cards, forms, layout specimens, logos.
.doc-block__code
Code surfaces, with optional toolbar and numbered lines.
.doc-block__meta
Notes, cautions, versioning and authoring guidance.
HTML 
<article class="doc-block doc-block--surface">
  <header class="doc-block__header">
    <h2 class="doc-block__title">Base Philosophy — Hybrid Reset</h2>
    <p class="doc-block__summary">One-paragraph overview.</p>
    <div class="doc-block__meta">
      <p class="doc-block__note--info">
        Short, repeatable note or caution.
      </p>
    </div>
  </header>

  <div class="doc-block__body section__stack">
    <div class="doc-block__preview">…example…</div>
    <div class="doc-block__code">…code…</div>
  </div>
</article>

Surfaces & density

Root modifiers control how strong each block feels on the page: neutral, surface, elevated, contrast, or compact.

Neutral

doc-block

Minimal padding, background inherited from section.

Surface

doc-block--surface

Own surface and border; ideal for primary explanatory blocks.

Elevated

doc-block--elevated

Stronger surface and subtle shadow; use for page intros.

Contrast / compact

doc-block--contrast doc-block--compact

High-contrast surface and tighter padding; good for callouts, warnings and “do / don’t” sections.

Code blocks, toolbar & numbered lines

Code surfaces share a common frame: toolbar, language label, actions and an optional numbered layout for step-by-step explanations.

Standard code block

CSS 
.doc-block__code {
  background: var(--color-code-bg);
  border-radius: var(--radius-md);
}

Use the toolbar only when there is real copy/expand behaviour wired via JS.

Numbered code (authoring)

Numbered code blocks are ideal for explaining multi-step snippets. Wrap each logical line in a child element; numbering is handled by CSS. 

:root {
  --layout-max-width: 80rem;  /* 1. Page container */
  --section-spacing-md: var(--space-7);  /* 2. Default vertical rhythm */
}

Any direct child of <code> becomes a numbered row: <span>, <div> or <code>. 

<div class="doc-block__code">
  <div class="doc-block__toolbar">
    <span class="doc-block__lang">HTML</span>
    <div class="doc-block__actions">
      <button class="doc-block__action" type="button">Copy</button>
      <button class="doc-block__action" type="button">Expand</button>
    </div>
  </div>

  <pre><code>...plain code...</code></pre>
</div>

<pre class="doc-block__code doc-block__code--numbered">
  <code>
    <span>/* 1. Step one */</span>
    <span>/* 2. Step two */</span>
  </code>
</pre>

Collapsed blocks & progressive disclosure

For long, advanced snippets, use .doc-block--collapsed with a toggle wired via JS. The CSS provides the collapsed visual state.

The CSS only handles the visual treatment. You are responsible for toggling .doc-block--collapsed in JavaScript.

Collapsed blocks are useful for implementation details that most readers can skip, but power users may expand.

CSS 
.doc-block--collapsed > .doc-block__body {
  max-height: var(--doc-collapsed-height, 18rem);
  overflow: hidden;
}

.doc-block--collapsed .doc-block__toolbar::after {
  content: "Collapsed";
}

Diff & change-log blocks

The diff pattern highlights additions and removals between two versions of a snippet. It is ideal for change logs and migration guides.

Visual behaviour

Lines marked as “added” or “removed” receive subtle inline chips and background cues. The rest of the code remains neutral. 

.button {
  padding-inline: var(--space-3);
  border-radius: var(--radius-full);
  background: #0070f3;
  background: var(--color-accent-soft);
}
<article class="doc-block doc-block--surface doc-block--diff">
  <header class="doc-block__header">
    <h2 class="doc-block__title">Buttons v2</h2>
  </header>

  <div class="doc-block__body">
    <div class="doc-block__code doc-block__code--numbered">
      <pre><code>
        <span>.button {</span>
        <span>  padding-inline: var(--space-3);</span>
        <span class="doc-block--diff-line-remove">
          background: #0070f3;
        </span>
        <span class="doc-block--diff-line-add">
          background: var(--color-accent-soft);
        </span>
        <span>}</span>
      </code></pre>
    </div>
  </div>
</article>

Semantic presets & highlights

Optional semantic modifiers make it easy to visually separate concepts, patterns, APIs and examples without inventing new ad-hoc styles.

Concept

Use for “what / why” explanations and mental models.

Pattern

Use for “how to structure” and repeatable layout or component patterns.

API

Use for class tables, props and token contracts.

Example

Use for fully wired HTML snippets and “copy-paste” recipes.

Accessibility & usage rules

Contract for accessible, consistent, token-driven documentation blocks. Maintain semantic structure, preserve toolbar interactions, and ensure code surfaces remain readable, navigable and keyboard-friendly.

Do

  • Use real semantic structure: <article class="doc-block">, .doc-block__header, .doc-block__body, .doc-block__preview, .doc-block__code.
  • Always include meaningful titles and summaries in .doc-block__header.
  • Use .doc-block__preview for visual examples and .doc-block__code for code; never mix roles.
  • Keep toolbar actions (Copy, Expand) keyboard accessible: they must remain actual <button> elements.
  • Use numbered blocks (.doc-block__code--numbered) for multi-step code explanations where each line needs a semantic row.
  • Use diff blocks (.doc-block--diff) to communicate additions/removals with semantic meaning.
  • Allow focus-visible to remain unmodified for all interactive elements in previews and toolbars.
  • Use prefers-reduced-motion fallback already embedded for accessibility; rely on tokens instead of custom transitions.

Don’t

  • Don’t wrap previews or code blocks in arbitrary containers; use only the provided block anatomy.
  • Don’t fake toolbar behaviour with non-button elements (<div> or <span>).
  • Don’t override padding, borders, backgrounds or radii inside .doc-block__preview or .doc-block__code.
  • Don’t hide essential information inside collapsed blocks without providing a clear toggle.
  • Don’t remove or override syntax colours — rely on --code-syntax-* tokens only.
  • Don’t embed screenshots where HTML specimens can be rendered directly.
  • Don’t mix semantic presets (.doc-block--concept, --pattern, --api, --example) in the same block. 
1. Use .doc-block anatomy exactly: header → body → preview/code → meta.
2. Keep toolbar actions semantic: real <button> with focus-visible.
3. Use previews for visuals and .doc-block__code for code (no mixing).
4. Apply numbered code only when each row has semantic meaning.
5. Use diff blocks for structural change explanation (add/remove).
6. Preserve token-driven spacing, borders, radii and motion.
7. Never remove focus-visible or syntax highlighting.
